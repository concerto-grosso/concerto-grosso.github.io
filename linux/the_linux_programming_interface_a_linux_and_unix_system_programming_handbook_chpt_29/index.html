《Linux/UNIX系统编程手册》第29章 线程：简介

发表于 2025-07-18 分类于 Linux 本文字数： 7.2k

　　与进程类似，线程是允许应用程序同时执行多个任务的机制之一。1个进程可以包含多个线程，这些线程独立地执行同一程序，并且共享全局内存 (包括未初始化数据段、已初始化数据段和堆)。实际上，每个线程的栈可能会与共享内存区域混合，具体取决于创建线程、加载共享库和附加共享内存区域的顺序。此外，线程栈的位置可能因Linux发行版而异。

　　属于同一进程的所有线程的栈都位于同一虚拟地址空间内，这表示可以通过合适的指针值来与其他线程共享数据。这种方法有时很有用，但需要仔细处理依赖关系，因为局部变量的有效性仅限于其所在的栈帧的生命周期。

　　传统的UNIX进程只是多线程进程的特殊情况之一，它是仅包含1个线程的进程。

　　线程可以并发执行。对于多处理器系统，多个线程可以同时执行。当某个线程阻塞在I/O时，其他线程可以继续执行。

　　对于某些应用程序，线程比进程更具优势。假设使用传统UNIX方法 (创建多个进程) 来实现并发，并以实现网络服务器为例。父进程接收来自客户端的连接请求，然后通过fork()创建子进程来处理与每个客户端的连接。虽然这种设计在很多情况下都很有效，但在部分应用程序中存在以下问题：

　　● 难以在进程间共享信息。由于父进程和子进程不会共享内存 (除了只读的文本段)，所以必须使用IPC机制来在进程间交换信息。

　　● 使用fork()创建进程的成本相对较高。即使使用了写时复制技术 (见24.1.2)，仍然需要复制页表和文件描述符表等进程属性，这表示fork()仍然比较耗时。

　　线程刚好解决了这2个问题：

　　● 线程间共享信息是简单且快速的，只需要将数据复制到共享 (全局和堆) 变量。但是，为了避免多个线程同时更新同一数据，需要使用同步机制。

　　● 线程创建比进程创建更快——通常至少快10倍 (对于Linux，线程是通过系统调用clone()实现的)，因为通过fork()创建子进程时必须复制的很多进程属性对线程是共享的 (不需要使用内存页的写时复制副本，也不需要复制页表)。

　　除了全局内存之外，线程还会共享很多属性 (这些属性对进程是全局，而非特定于线程的)，包括进程ID、父进程ID、进程组ID、会话ID、控制终端、进程凭证 (用户和组ID)、打开的文件描述符、通过fcntl()创建的记录锁、信号处理方式、文件系统相关属性 (umask、当前工作目录和根目录)、间隔定时器 (setitimer())、POSIX定时器 (timer_create())、System V信号量撤销 (semadj) 值、资源限制、使用的CPU时间 (由times()返回)、使用的资源 (由getrusage()返回) 和nice值 (由setpriority()和nice()设置)。

　　每个线程特有的属性包括线程ID、信号掩码、特定于线程的数据、备用信号栈、变量errno、浮点环境、实时调度策略和优先级、CPU亲和性 (特定于Linux)、能力 (特定于Linux) 以及栈。

　　当启动程序后，结果进程仅会包含1个线程，称为初始线程或主线程。

　　线程的执行会通过以下方式终止：

　　● 线程启动函数通过return指定返回值返回。

　　● 线程调用pthread_exit()。

　　● 线程被pthread_cancel()取消。

　　● 任一线程调用exit()或主线程 (在主函数中) 返回，这会导致进程的所有线程立即终止。

29.1 Pthreads API的背景

　　在20世纪80年代晚期和20世纪90年代早期，出现了多种线程API。在1996年，POSIX.1c标准化了POSIX线程API，该标准随后被纳入了SUSv3。

29.1.1 Pthreads数据类型

　　Pthreads API定义了大量数据类型，下表列出了其中的部分：

数据类型描述
pthread_t线程ID
pthread_mutex_t互斥锁
pthread_mutexattr_t互斥锁属性对象
pthread_cond_t条件变量
pthread_condattr_t条件变量属性对象
pthread_key_t特定于线程的数据的键值
pthread_once_t一次性初始化控制上下文
pthread_attr_t线程属性对象

　　SUSv3没有指定如何表示这些数据类型，可移植应用程序应该将其视为不透明数据。因此，程序应该避免依赖于这些数据类型的变量结构或内容 (即不能使用运算符==来比较这些数据类型的变量)。

29.1.2 线程和errno

　　对于传统UNIX API，errno是全局整数变量，但这对多线程程序还不够。若某个线程调用的函数会通过全局变量errno返回错误，则可能导致其他调用此类函数并检查errno的线程出错，即产生竞争条件。因此，对于多线程程序，每个线程有自己的errno值。Linux实现特定于线程的errno的方式与大多数UNIX实现类似：将errno定义为宏，它展开为1个函数调用，返回1个对每个线程都不同的可修改的左值 (由于左值是可修改的，所以在多线程程序中可以使用赋值语句errno = value)。

　　综上所述，errno机制已适应线程，并且报告错误的方式与传统UNIX API相同。

　　最初的POSIX.1遵循K&R的C用法，允许程序将errno声明为extern int errno。SUSv3不允许这种用法 (该变更实际发生在1995年的POSIX.1c中)。现在，程序需要通过引用errno.h来声明errno，这使得每个线程都有自己的errno。

29.1.3 Pthreads函数的返回值

　　系统调用和部分库函数返回状态的传统方法是成功时返回0，出错时返回-1并设置errno。Pthreads API中的函数返回状态的方法有所不同。所有Pthreads函数都是成功时返回0，出错时返回正值，该值与传统UNIX系统调用设置给errno的值相同。

29.1.4 编译多线程程序

　　对于Linux，使用Pthreads API的程序在编译时必须带有选项cc –pthread，该选项具有以下作用：

　　● 定义预处理器宏_REENTRANT，这会暴露一些可重入函数。

　　● 程序会与库libpthread一起链接 (等同于-lpthread)。

　　实际上，编译多线程程序的选项因UNIX实现 (和编译器) 而异。Tru64等UNIX实现与Linux相同，而Solaris和HP-UX使用的是cc –mt。

29.2 线程创建

　　函数pthread_create()会创建1个新线程，新线程从调用参数start指向的函数开始，并且会带有参数arg。调用pthread_create()的线程会继续执行 (这与系统调用clone()的glibc包装函数的行为相同)。

　　参数arg声明为void *，这表示可以传递任何类型的对象。arg通常会指向全局或堆变量，有时也会被指定为NULL。若需要传递多个参数，则可以将arg指向结构体，该结构体的成员就是这些参数。通过类型转换，甚至可以将arg指定为int。start指向的函数的返回值也是void *，所以它可以像arg那样使用。

　　严格来说，C标准并未定义将void *转换为int以及将int转换为void *的结果。但是，大多数C编译器允许这些操作，并且会产生预期的结果。

　　参数thread指向pthread_t类型的缓冲区，在pthread_create()返回之前，会将该线程的ID复制到该缓冲区中 (该ID用于区分属于同一进程的不同线程，可以在后续的Pthreads调用中引用该线程)。SUSv3明确规定，在新线程开始执行之前，实现无需初始化thread指向的缓冲区，这表示新线程可能在pthread_create()返回之前开始执行。若新线程需要获取自己的ID，则必须调用pthread_self()。

　　参数attr指向pthread_attr_t对象，用于指定新线程的各种属性。若attr为NULL，则创建线程时将使用默认属性。

　　这个示例展示了线程属性结构体的用法。首先，将线程属性结构体初始化为默认值 (第5~9行) 并设置分离状态 (第11~15行)。然后，使用该结构体创建线程 (第17~21行)。一旦线程创建完成，该结构体不再需要，所以将其销毁 (第23~27行)。

　　在调用pthread_create()后，程序无法保证下一个将被调度使用CPU的线程是哪一个 (对于多处理器系统，可以有多个线程同时在不同的CPU上执行)。

29.3 线程终止

　　函数pthread_exit()会终止调用线程。参数retval用于指定返回值，其他线程可以通过pthread_join()来获取该返回值。

　　调用pthread_exit()等同于在线程启动函数中执行返回，不同的是pthread_exit()可以由线程启动函数调用的函数调用。

　　retval指向的值不应该位于栈中，因为线程终止后栈会被回收。同样，指定给线程启动函数的return语句的值也不应该位于栈中。

　　若主线程调用pthread_exit() (而非exit()或return语句)，则其他线程会继续执行。

29.4 线程ID

　　函数pthread_self()会返回调用线程的线程ID。函数pthread_equal()会判断线程IDt1和t2是否相对。

　　pthread_equal()是必要的，因为pthread_t必须被视为不透明数据。SUSv3不要求pthread_t被实现为标量类型。对于Linux，该数据类型定义为unsigned long。但是，对于有些UNIX实现，该数据类型可能定义为指针或结构体 (对于NPTL，pthread_t实际上是已经转换为unsigned long的指针)。

　　对于Linux线程实现，线程ID在进程之间是唯一的，但有些UNIX实现并非如此。SUSv3明确规定，应用程序不应该使用同一线程ID来识别不同进程中的线程。SUSv3还规定，在终止的线程被pthread_join()合并或分离线程终止后，线程ID可以重用。

　　POSIX线程ID不同于特定于Linux的系统调用gettid()返回的线程ID。POSIX线程ID由线程实现分配和维护。gettid()返回的线程ID是由内核分配的整数 (与进程ID类似)。对于Linux NPTL线程实现，尽管每个POSIX线程有唯一的内核线程ID，但应用程序通常不需要知道该ID (若应用程序依赖于该ID，则会导致其无法被移植)。

29.5 合并终止的线程

　　函数pthread_join()会等待线程ID为thread的线程终止，该操作称为合并。若目标线程已终止，则该函数会立即返回。

　　若参数retval不为NULL，则它会收到终止的线程的返回值，即线程执行返回或调用pthread_exit()时指定的值。

　　调用pthread_join()时将thread指定为已合并的线程的线程ID会导致无法预测的行为 (例如，将随后创建的重用该ID的新线程合并)。

　　若线程未分离，则必须通过pthread_join()来合并它。若不这样做，则线程终止后会转换为僵尸线程。除了浪费系统资源之外，若积累了足够多的僵尸线程，则会导致无法创建新线程。

　　pthread_join()对线程执行的操作与waitpid()对进程执行的操作类似，但存在以下主要区别：

　　● 线程是相互对等的。任何线程都可以调用pthread_join()来合并进程中的其他线程 (例如，若线程A创建了线程B，线程B创建了线程C，则线程A可以合并线程C，反之亦然)。这不同于进程之间的层级关系。当父进程通过fork()创建子进程后，只有父进程能够通过waitpid()等待子进程。

　　● 必须明确指定待合并的线程 (对于进程，可以通过waitpid(-1. &status, options)来等待任意进程)，并且也无法进行非阻塞合并 (对于进程，可以通过指定标志WNOHANG来进行非阻塞等待)。此外，也无法实现类似条件变量的功能。

　　pthread_join()只能与指定线程合并是故意的，其理念是线程应该仅与 "已知" 的线程合并。"与任意线程合并" 操作的问题在于线程之间没有层级，这表示该操作可以与任何线程合并，包括由库函数私自创建的线程。因此，库函数将无法合并线程来获取其状态，并且会错误地尝试合并已合并的线程。

29.6 分离线程

　　线程默认处于可合并状态，这表示当线程终止时，其他线程可以通过pthread_join()来获取其返回状态。有时，可能只需要系统在线程终止时自动清理并删除线程。为此，可以通过调用pthread_detach()并将参数thread指定为线程ID来将线程标记为分离状态。一旦如此，将无法通过pthread_join()来获取其返回状态，并且该线程无法再回到可合并状态。

　　线程可以调用pthread_detach(pthread_self())来分离自己。

　　分离线程依然会受到其他线程调用exit()或主线程返回的影响。对于这些情况，进程的所有线程都会立即终止，无论处于可合并状态还是分离状态。换句话说，pthread_detach()仅控制线程终止后的发生的事情，而非线程如何 (或何时) 终止。
