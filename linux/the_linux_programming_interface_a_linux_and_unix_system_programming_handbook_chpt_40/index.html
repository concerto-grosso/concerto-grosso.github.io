《Linux/UNIX系统编程手册》第40章 登录统计

发表于 2025-12-14 更新于 2025-12-15 分类于 Linux 本文字数： 13k

　　UNIX系统维护了2个包含用户登录和注销相关信息的文件：utmp和wtmp。

　　● 文件utmp维护着当前登录的用户的记录等信息。当用户登录时，1条记录会被写入到该文件。该记录的字段ut_user记录用户的登录名，并且该记录会在用户注销时被删除。命令who就是通过该文件的信息来显示当前登录用户的列表。

　　● 文件wtmp是所有用户登录和注销的审查记录。当用户登录时，除了会向文件utmp写入1条记录之外，也会向该文件写入1条相同的记录。当用户注销时，还会向该文件写入1条记录。用户注销时写入的记录与用户登录时写入的信息相同，除了字段ut_user会被清空。命令last可用于显示和筛选文件wtmp的内容。

　　对于Linux，文件utmp和wtmp的绝对路径名分别是/var/run/utmp和/var/log/wtmp。通常，应用程序无需知道这些路径名，因为它们会被编译到glibc。需要使用这些路径名的程序应该使用头文件paths.h和utmpx.h中定义的常量_PATH_UTMP和_PATH_WTMP (而非显式地将路径名编码到程序中)。

　　SUSv3没有标准化文件utmp和wtmp的路径名的符号名。Linux和BSD使用_PATH_UTMP和_PATH_WTMP。很多其他UNIX实现将这2个路径名定义为UTMP_FILE和WTMP_FILE，Linux在头文件utmp.h中也定义了这2个符号名。

　　文件utmp和wtmp自UNIX早期就存在，但在各种UNIX实现中经历了持续的演化和分化 (尤其是BSD与System V之间)。System V Release 4极大地扩展了API，在该过程中创建了新的 (并行) 结构体utmpx以及相关文件utmp和wtmp。头文件和用于处理这些新文件的附加函数的名称也包括字母x。很多其他UNIX实现也向API中添加了自己的扩展。

　　Linux的utmpx API是BSD实现和System V实现的混合体。Linux不像System V那样创建并行的文件utmp和wtmp。相反，这2个文件包含了所有必需的信息。但是，为了兼容UNIX实现，Linux同时提供了传统的utmp API以及System V衍生的用于访问这些文件内容的utmpx API。对于Linux，这2个API的信息完全相同 (两者的区别之一是前者提供了部分函数的可重入版本，而后者未提供)。此外，SUSv3仅说明了utmpx API，所以考虑与UNIX实现兼容时该API是首选。但是，SUSv3没有说明utmpx API的所有方面 (例如，文件utmp和wtmp的路径名)，所以登录统计文件在不同的实现中略有不同，并且各种实现提供了SUSv3说明的额外登录统计函数。

40.1 结构体utmpx

　　文件utmp和wtmp由utmpx记录组成。utmpx记录对应的结构体utmpx被定义在头文件utmpx.h中。

　　结构体utmpx的所有字符串成员都是以字符\0结尾，除非它被完全填满。

　　对于登录进程，成员ut_line和ut_id存储的信息源于终端设备的名称。成员ut_line包含终端设备的完整文件名。成员ut_id包含文件名的后缀部分——tty、pts (System V风格的伪终端) 或pty (BSD风格的伪终端) 后的字符串 (例如，对于终端/dev/tty2，ut_line和ut_id分别是tty2和2)。SUSv3详细说明了这2个成员，但未说明了它们的长度。

　　对于窗口化环境，有些终端模拟器会使用成员ut_session来记录终端窗口的会话ID。

　　SUSv3未说明成员ut_host、ut_exit、ut_session和ut_addr_v6。大多数其他实现都提供了ut_host和ut_exit，部分其他实现提供了ut_session，ut_addr_v6是特定于Linux的。

　　成员ut_type是整数，用于定义写入到文件的记录的类型。该成员可以被指定为以下常量之一 (对应的值在后面的括号中)：

　　● EMPTY (0)：该记录不会包含有效的统计信息。

　　● RUN_LVL (1)：该记录表示系统在启动或关闭期间运行级别发生了变化。当定义了宏_GNU_SOURCE后，该常量的定义才会出现在头文件utmpx.h中。

　　● BOOT_TIME (2)：该记录在字段ut_tv中包含系统引导时间。记录RUN_LVL和BOOT_TIME的写入者通常是进程init。这些记录会被同时记录到文件utmp和wtmp。

　　● NEW_TIME (3)：该记录在字段ut_tv中包含系统时钟更改后的新时间。

　　● OLD_TIME (4)：该记录在字段ut_tv中包含系统时钟更改前的旧时间。当NTP等守护进程修改系统时钟时，它会将类型为NEW_TIME和OLD_TIME的记录会被写入到文件utmp和wtmp。

　　● INIT_PROCESS (5)：该记录是进程init生成的进程 (例如，进程getty) 的记录。

　　● LOGIN_PROCESS (6)：该记录是用户登录的会话组长进程 (例如，进程login) 的记录。

　　● USER_PROCESS (7)：该记录是用户进程的记录 (通常是登录会话)，用户名存储在字段ut_user中。登录会话可能由进程login启动，也可能由提供远程登录的工具 (例如，ftp和ssh) 启动。

　　● DEAD_PROCESS (8)：该记录标识已退出的进程。

　　类型为INIT_PROCESS的记录通常对应getty (或类似的程序) 的执行。当系统引导时，进程init会为每个终端行和虚拟控制台创建1个子进程，每个子进程会运行程序getty。程序getty会打开终端，提示用户登录名，然后执行login。当成功验证用户和执行其他步骤后，login会创建1个子进程来执行用户的登录shell。这种登录会话的完整生命周期由写入到文件wtmp的4条记录表示，顺序如下：

　　● 进程init写入的类型为INIT_PROCESS的记录；

　　● 程序getty写入的类型为LOGIN_PROCESS的记录；

　　● 程序login写入的类型为USER_PROCESS的记录；

　　● init检查到子login进程终止 (用户注销) 时写入的类型为DEAD_PROCESS的记录。

　　某些版本的进程init在更新文件wtmp之前启动程序getty。因此，两者会争相更新文件wtmp，导致类型为INIT_PROCESS和LOGIN_PROCESS的记录的写入顺序可能与这里的顺序相反。

40.2 从文件utmp和wtmp中检索信息

　　本小节介绍的函数会从包含utmpx格式记录的文件中检索记录。这些函数默认会使用文件utmp，但这可以通过函数utmpxname()修改。

　　这些函数使用了文件中的当前位置 (类似于文件偏移量) 的概念，从该位置检索记录。这些函数都会更新这个位置。

　　函数setutxent()会将当前位置重置到文件开头。通常，在检索记录前应该调用该函数。这可以防止调用的第三方函数之前调用过的函数getutx*()而可能导致的混淆。根据正在执行的任务，可能还需要在程序后续的适当位置再次调用setutxent()。

　　函数setutxent()和getutx*()都会在文件utmp未打开时打开该文件。当不再需要使用该文件时，应该调用函数endutxent()来关闭它。

　　函数getutxent()、getutxid()和getutxline()会从文件utmp读取记录并返回指向静态分配的结构体utmpx的指针。

　　函数getutxent()会从文件utmp中按序检索下一条记录。

　　函数getutxid()会根据参数ut指向的结构体utmpx的成员ut_type和ut_id从文件utmp中检索记录：

　　● 若成员ut_type是RUN_LVL、BOOT_TIME、NEW_TIME或OLD_TIME，则getutxid()会搜索字段ut_type匹配指定值的下一条记录 (这些类型的记录与用户登录无关)。这允许搜索系统时间修改和运行级别修改的记录。

　　● 若成员ut_type是INIT_PROCESS、LOGIN_PROCESS、USER_PROCESS或DEAD_PROCESS，则getutxid()会搜索字段ut_type匹配这些值中的任意1个且字段ut_id匹配参数ut中的指定值的下一条记录。这允许搜索与特定终端对应的记录。

　　函数getutxline()会从文件utmp中检索字段ut_type为LOGIN_PROCESS或USER_PROCESS且字段ut_line匹配参数ut中的指定值的下一条记录。这对查找与用户登录对应的记录很有用。

　　函数getutxid()和getutxline()在搜索失败 (在未找到记录之前遇到EOF) 时会返回NULL。

　　对于有些UNIX实现，getutxid()和getutxline()会将用于返回结构体utmpx的静态区域视为缓存。若这2个函数确定之前的getutx*()调用放入该缓存中的记录与ut中的指定条件匹配，则不会进行文件读取，而是再次返回相同的记录 (SUSv3允许这种行为)。因此，为了防止在循环中调用这2个函数时重复返回同一条记录，必须将该静态区域清零，这可以通过以下代码实现：

　　因为getutx*()会返回指向静态分配的结构体的指针，所以这些函数是不可重入的。glibc提供了传统utmp函数的可重入版本 (getutent_r()、getutid_r()和getutline_r())，但未提供传统utmp函数的对应utmpx函数的可重入版本 (SUSv3没有说明这些可重入函数)。

　　在默认情况下，所有的getutx*()都作用于文件utmp。若需要使用其他文件 (例如，文件wtmp)，必须先调用函数utmpxname()来指定目标路径名。

40.3 getlogin()

　　函数getlogin()会返回在调用进程的控制终端登录的用户的名称。该函数使用的是文件utmp中维护的信息。

　　getlogin()会调用ttyname()来查找与调用进程的标准输入关联的控制终端的名称，然后在文件utmp中搜索字段ut_line与该终端名匹配的记录。若找到，则返回该记录的ut_user。若未找到或出错，则返回NULL并设置error。导致该函数出错的原因之一是进程没有与标准输入关联的终端 (ENOTTY)，这可能是因为该进程是守护进程。另一种可能性是文件utmp中没有记录该终端会话 (例如，某些软件终端模拟器不会在文件utmp中创建条目)。

　　即使用户ID在密码文件中有多个登录名，getlogin()还是能够返回用于在该终端登录的用户名，因为该函数依赖于文件utmp。相比之下，调用getpwuid(getuid())总是从密码文件中检索第1条匹配的记录。

　　SUSv3详细说明了getlogin()的可重入版本getlogin_r()，glibc也提供了该函数。

　　环境变量LOGNAME也能用于获取用户的登录名。但是，该环境变量的值可以被用户修改，所以该环境变量不能安全地标识用户。

40.4 更新登录会话的文件utmp和wtmp

　　当开发创建登录会话的应用程序时，应该按照以下方式更新文件utmp和wtmp：

　　● 当登录时，应该向文件utmp中写入1条记录来表示该用户。应用程序必须检查文件utmp中是否已存在该终端的记录。若存在，则覆盖；否则，在该文件的末尾写入1条新记录。通常，调用pututxline()足以确保这会正确执行。结果记录应该至少包含字段ut_type、ut_user、ut_tv、ut_pid、ut_id和ut_line。字段ut_type应该被设置为USER_PROCESS。字段ut_id应该包含用户登录的设备 (终端和伪终端) 的名称后缀。字段ut_line应该包含登录设备的名称，并删除开头的字符串/dev/。相同的记录也会被写入到文件wtmp的末尾。文件utmp中的记录的唯一标识是终端名 (字段ut_line和ut_id的组合)。

　　● 当注销时，之前向文件utmp中写入的记录应该被删除。这能通过创建1条字段ut_type为DEAD_PROCESS，字段ut_user为空，其他字段与登录时写入的记录相同的记录来完成。该记录会覆盖之前的记录，并且也会被写入到文件wtmp的末尾。

　　若无法在注销时清理文件utmp中的记录 (可能因为程序崩溃)，则下一次重启时进程init会自动将该记录的字段ut_type设置为DEAD_PROCESS，并清空各种其他字段来清理该记录。

　　文件utmp和wtmp通常受到保护，所以更新它们需要特权。函数getlogin()的准确性取决于文件utmp的完整性。出于这个原因和其他原因，文件utmp和wtmp绝不应被设置为允许非特权用户写入。

　　除了通过login、telnet和ssh的登录会被记录在登录统计文件之外，大多数ftp实现也会创建登录统计记录。但是，系统中启动的所有终端窗口是否都会创建登录记录因UNIX实现而异。

　　对于某些终端模拟器程序 (例如，xterm)，命令行选项或其他机制可用于决定程序是否会更新登录统计文件。

　　函数pututxline()会将参数ut指向的结构体utmpx对应的记录写入到文件utmp (或调用utmpxname()指定的文件)。

　　在写入记录之前，pututxline()会先通过getutxid()来搜索待更新的记录。若找到，则更新；否则，向文件的末尾写入1条新记录。在很多情况下，应用程序在调用pututxline()之前会先调用getutx*()来将当前位置设置为目标记录的位置。若pututxline()检查到已经发了这种情况，则不会调用getutxid()。

　　当pututxline()内部调用getutxid()时，该调用不会修改getutx*()用于返回结构体utmpx的静态区域。这也是SUSv3所要求的。

　　函数updwtmpx()会将参数ut指向的结构体utmpx对应的记录写入到参数wtmpx_file指定的文件。SUSv3没有说明该函数，并且只有部分UNIX实现提供了该函数。其他实现提供了类似的函数 (例如，login()、logout()和logwtmp())。

40.5 文件lastlog

　　文件lastlog记录了每个用户上一次登录的时间。该文件的作用之一是允许登录程序在用户开始新登录会话时告知它们上一次登录的时间。除了更新文件utmp和wtmp之外，提供登录服务的应用程序也应该更新该文件。

　　与文件utmp和wtmp相同，文件lastlog的位置和格式在各个实现之间也存在差异 (有些UNIX实现未提供该文件)。对于Linux，该文件的位置是/var/log/lastlog，并且头文件paths.h中定义的常量_PATH_LASTLOG指向该位置。与文件utmp和wtmp类似，文件lastlog通常受到保护，所以只有特权进程才能更新该文件 (所有进程都能读该文件)。

　　文件lastlog中的记录对应的结构体lastlog被定义在头文件lastlog.h中。

　　文件lastlog中的记录不包含用户名或用户ID。相反，该文件由一系列按用户ID索引的记录组成。因此，为了查找用户ID为1000的记录，需要寻找该文件的字节(1000 * sizeof(struct lastlog))。这也会导致无法区分同一用户ID的不同用户名。
