《Linux/UNIX系统编程手册》第55章 文件加锁

发表于 2026-06-24 更新于 2026-06-26 分类于 Linux 本文字数： 11k

　　常见的应用程序需求之一是从文件读取数据、修改这些数据并写回到文件。只要同时只有1个进程以这种方式操作文件时，就不会产生问题。但是，当多个进程同时更新同一文件时，就会产生问题。假设进程A和B都会通过以下方式更新包含序列号的文件：

　　1) 从文件读取序列号。

　　2) 将序列号用于由应用程序定义的目的。

　　3) 增加序列号并写回到文件。

　　由于缺乏同步机制，这可能导致竞争条件。尽管信号量可用于执行同步，但文件锁更可取，因为内核会自动将锁与文件关联。

　　本章将介绍2个用于设置文件锁的系统调用：

　　● flock()会为整个文件加锁 (源于BSD)。

　　● fcntl()会为文件的指定区域加锁 (源于System V)。

　　这2个系统调用的用法通常都是为文件加锁、执行文件I/O并解锁文件 (它们创建的锁之间的相互作用是未定义的，所以应用程序应该对同一文件仅使用其中1种加锁方式)。

　　尽管文件加锁通常与文件I/O一起使用，但它也能作为更通用的同步技术。协调进程的行为可以遵循惯例：锁定文件的全部或部分表示进程正在访问除该文件以外的某些共享资源 (例如，共享内存)。

　　因为stdio库会执行用户空间缓冲，所以使用本章介绍的文件加锁技术时应该谨慎使用stdio函数。问题在于输入缓冲区在加锁之前可能被填满或输出缓冲区在解锁后可能被刷新。可以通过以下方法避免这些问题：

　　● 使用read()和write() (或类似的系统调用) 执行文件I/O (而非stdio库)。

　　● 在锁定文件之后立即刷新stdio流，并在解锁文件之前再次刷新stdio流。

　　● 尽管会牺牲一些效率，但可以通过setbuf() (或类似的函数) 来禁用stdio缓冲。

　　本章会将文件锁分类为建议锁和强制锁。文件锁默认是建议锁，进程可以忽略其他进程设置的建议锁。强制锁会强制执行I/O的进程遵循其他进程持有的锁 (SUSv3没有说明强制锁，但大多数现代UNIX实现都提供了强制锁)。

　　文件租约是Linux 2.4或更高版本中提供的Linux特有机制。当某个进程获取了文件租约后，若其他进程试图对该文件调用open()或truncate()，则该进程会收到作为通知的信号 (包含truncate()是必要的，因为它是唯一可以在不打开文件的情况下修改文件内容的系统调用)。文件租约是为了让Samba支持Microsoft SMB协议的机会锁 (oplock) 功能以及让NFS第4版支持委托机制 (类似于机会锁)。

55.1 flock()

　　尽管fcntl()提供了flock()所提供的功能的超集，但flock()仍然用于某些应用程序，因为其继承和解锁的某些语义不同于fcntl()。

　　系统调用flock()会为整个文件加锁或解锁，目标文件由文件描述符fd标识。参数operation用于控制该函数的行为，必须包含值LOCK_SH、LOCK_EX或LOCK_UN之一。

operation值描述
LOCK_SH为fd引用的文件设置共享锁
LOCK_EX为fd引用的文件设置独占锁
LOCK_UN解锁fd引用的文件
LOCK_NB以非阻塞方式加锁

　　当其他进程持有文件的不兼容锁时，flock()默认会阻塞进程。为了防止这种情况，可以在operation中指定LOCK_NB (或运算|)。此时，flock()导致错误EWOULDBLOCK。

　　若干个进程可能同时持有文件的共享锁。但是，同时只能有1个进程持有文件的独占锁 (换句话说，独占锁会阻止其他进程的共享锁和独占锁)。

　　无论文件的访问模式 (只读、只写或读写) 如何，进程都可以对文件加锁。

　　可以通过调用flock()并将operation指定为合适的值来将现有共享锁转换为独占锁 (反之亦然)。当进程将共享锁转换为独占锁时，若其他进程持有同一文件的共享锁，则该进程会被阻塞 (除非指定了LOCK_NB)。

　　锁的转换过程不是原子的。在转换过程中，现有锁会被删除，然后设置新锁。在此期间，其他进程可能请求不兼容的锁。在这种情况下，转换将会被阻塞，若指定了LOCK_NB，则转换将出错并失去原有的锁。这种行为在最初的BSD实现中就存在，并且在很多其他UNIX实现中也存在。

　　尽管SUSv3没有说明flock()，但大多数UNIX实现都提供了该系统调用。有些实现需要引用头文件fcntl.h或sys/fcntl.h (而非sys/file.h)。因为该系统调用源于BSD，所以该系统调用设置的锁也称为BSD文件锁。

55.1.1 锁继承和释放的语义

　　flock()可用于释放锁 (操作LOCK_UN)。此外，锁会在对应文件描述符被关闭时自动被释放。但是，实际情况比这更复杂。通过flock()获取的锁会与打开的文件描述关联 (而非文件描述符或i-node)。这表示复制文件描述符时新文件描述符会引用同一文件锁。

　　当通过特定文件描述符获取锁并复制该文件描述符后，若没有显式地解锁，则锁只会在所有副本文件描述符都被关闭时才会被释放。

　　当通过open()获取同一文件的第2个文件描述符 (或关联的文件描述) 时，flock()会将第2个文件描述符独立处理。

　　当通过fork()创建子进程时，子进程会继承父进程的文件描述符。父子进程的文件描述符会引用同一打开的文件描述 (包括锁)。可以通过该语义来巧妙地将文件锁从父进程转移给子进程，从而使该锁完全由子进程控制 (当通过fork()创建子进程后，父进程关闭其文件描述符)。

　　通过flock()创建的锁会在exec()期间得到保留 (除非启用了文件描述符的close-on-exec标志，并且该文件描述符是最后1个引用底层打开的文件描述的文件描述符)。

　　上述Linux语义与flock()的传统BSD实现一致。某些UNIX实现将flock()实现为基于fcntl() (其语义不同于fcntl())。

55.1.2 flock()的限制

　　使用flock()加锁时存在以下限制：

　　● 只能对整个文件加锁。这种粗略的加锁方式会限制协作进程之间的并发性 (例如，当多个进程试图访问同一文件的不同部分时，使用flock()会不必要地阻止这些进程并发运行)。

　　● flock()只能设置建议锁。

　　● 很多NFS实现不识别通过flock()设置的锁。

　　从历史角度，Linux NFS服务器不支持flock()锁。从内核2.6.12开始，Linux NFS服务器支持flock()锁，其实现方式是对整个文件使用fcntl()锁。当客户端和服务器都使用BSD锁时，可能会出现奇怪的现象：客户端通常看不到服务器的锁，反之亦然。

55.2 fcntl()

　　fcntl()可用于对文件的任意部分加锁，从1个字节到整个文件。这种锁定的方式称为记录锁定。但是，这个术语并不准确，因为UNIX系统中的文件是字节序列，没有记录边界的概念。文件中任何记录的概念都完全是由应用程序内部定义的。

　　术语记录锁定源于fcntl()通常用于锁定文件中与应用程序定义的记录边界对应的字节范围。字节范围、文件区域和文件段这些术语的使用频率更低，但更准确地描述了这种类型的锁定 (这是POSIX.1和SUSv3中唯一说明的锁类型，所以有时也称为POSIX文件锁定)。

　　SUSv3要求常规文件必须支持记录锁，并允许其他类型的文件支持记录锁。但是，Linux允许将其应用于任何类型的文件描述符。

55.2.1 结构体flock

　　结构体flock的成员l_type定义了锁的类型，可以为值F_RDLCK (读取锁)、F_WRLCK (写入锁) 或F_UNLCK (解锁)。

　　从语义角度，fcntl()的读取锁和写入锁对应flock()的共享锁和独占锁，因为两者的兼容性规则相同：多个个进程都可以持有同一文件区域的读取锁，而只有1个进程能够持有文件区域的写入锁，并且写入锁会排斥其他读取锁和写入锁，F_UNLCK类似于flock()的操作LOCK_UN。

　　对文件设置读取锁需要以读取模式打开该文件，对文件设置写入锁需要以写入模式打开该文件。对文件设置读取锁和写入锁以读写模式打开该文件。试图设置与访问模式不匹配的锁会导致错误EBADF。

　　成员l_whence、l_start和l_len用于指定待锁定的字节范围 (前2个成员与lseek()的参数whence和offset类似，l_start指定如何解释文件内的偏移量)。

　　● l_whence为SEEK_SET时表示将l_start解锁为相对文件开头的字节数。

　　● l_whence为SEEK_CUR时表示将l_start解锁为相对文件当前偏移量的字节数。

　　● l_whence为SEEK_END时表示将l_start解锁为相对文件末尾的字节数。

　　对于后2种情况，l_start可以为负值，只要结果文件位置不在文件起始位置 (字节0) 之前即可。

　　l_len用于指定待锁定的字节数。可以锁定文件末尾之后的字节，但无法锁定文件开头之前的字节。从内核2.4.21开始，Linux允许将该成员指定为负值，这表示锁定字节l_start - abs(l_len)到l_start - 1。SUSv3允许但不要求提供该特性，但部分UNIX实现提供了该特性。

　　应用程序通常应锁定必要的最小字节范围。这样可以提高其他进程同时试图锁定同一文件不同区域的并发性。在某些情况下，"最小范围" 这一术语需要加以限定。对于NFS和CIFS等网络文件系统，混合使用记录锁和mmap()可能导致问题。因为mmap()以系统页大小为单元映射文件，所以当文件锁不是按页对齐时，会产生竞争条件：当映射页的任何部分被修改时，内核可能会写入锁未覆盖的区域。

　　将l_len指定为0表示锁定从l_whence和l_start指定的位置到文件结尾的所有字节。为了锁定整个文件，可以将l_whence指定为SEEK_SET，并将l_start和l_len都指定为0。

55.2.2 参数cmd

　　当用于文件锁，fcntl()的参数cmd可以被指定为操作F_SETLK、F_SETLKW或F_GETLK。

　　● 操作F_SETLK表示锁定 (l_type为F_RDLCK或F_WRLCK) 或解锁 (l_type为F_UNLCK) 指定字节范围。若其他进程持有该字节范围的任意部分的不兼容锁，则fcntl()会导致错误EAGAIN。对于某些UNIX实现，fcntl()会在这种情况下会导致错误EACCES。SUSv3允许这2种可能性，可移植应用程序应该判断这2种情况。

　　● 操作F_SETLKW与F_SETLK类似，但当其他进程持有待锁定字节范围的任意部分的不兼容锁时，fcntl()会阻塞进程。若涉及信号处理，并且未指定标志SA_RESTART，则fcntl()可能会被中断并导致错误EINTR。该特性可用于为锁请求设置超时时间 (alarm()或setitimer())。

　　● 操作F_GETLK表示仅检查是否可以锁定指定字节范围 (成员l_type必须为F_RDLCK或F_WRLCK)。flockstr会被视为值-结果参数，fcntl()会通过该参数来返回检查结果。若可以加锁，则l_type会被设置为F_UNLCK，其他成员不变。若指定字节范围存在不兼容锁，则flockstr会返回其中某个锁的信息 (无法确定是哪个锁)，包括其类型 (l_type)、字节范围 (l_start和l_len，而l_whence总是会被设置为SEEK_SET) 以及持有该锁的进程的进程ID (l_pid)。

　　fcntl()要么锁定指定的整个字节范围，要么完全不锁定。该系统调用没有锁定指定范围中未被锁定的字节的概念。

　　当在F_GETLK后使用F_SETLK或F_SETLKW时，可能会产生竞争条件 (F_GETLK返回的信息可能已过时)。因此，F_GETLK可能并没有最初看起来那样有用。

　　glibc还提供了fcntl()的包装函数lockf() (SUSv3详细说明了该函数，但没有说明它与fcntl()的关系。尽管如此，大多数UNIX实现仍然将其实现为基于fcntl())。lockf(fd, operation, size)调用等同于l_whence为SEEK_CUR、l_start为0、l_len为size的fcntl()调用。operation类似于fcntl()的参数cmd，但加锁、解锁和检查锁时使用的是不同的值。此外，lockf()只会设置独占 (写入) 锁。

55.2.3 加锁和解锁的细节

　　关于通过fcntl()加锁和解锁，有以下几点值得注意：

　　● 解锁文件区域总是会立即成功。解锁未持有锁的文件区域不会导致错误。

　　● 进程同时只能最多持有特定文件区域的1种类型的锁，当再次锁定已锁定的区域时，要么不会发生任何变化 (目标锁的类型与现有锁的类型相同)，要么会以原子方式将锁转换为新模式。对于后者，当将读取锁转换为写入锁时，需要考虑该调用出错 (F_SETLK) 或阻塞进程 (F_SETLKW) 的情况。

　　● 进程不会因为自己持有的锁而被阻塞 (即使通过引用同一文件的不同文件描述符)。

　　● 当进程以不同的锁模式锁定其持有的锁的中间部分时，会产生3个锁：在新锁的两端会产生2个与原锁相同的较小的锁。相反，若以相同的锁模式获取与现有锁重叠或相邻的第2个锁，则会导致这2个锁合并，合并后的锁覆盖2个锁的合并区域。若新锁与模式不同的现有锁重叠，则现有锁会收缩，因为重叠的部分会被纳入到新锁。

　　● 关闭文件描述符在文件区域锁方面有一些特殊的语义 (见55.2.8)。

55.2.4 死锁

　　使用操作F_SETLKW时需要注意上图所示的死锁：进程A和B都被阻塞在第2个锁请求。若内核不加以检查，这2个进程会永远被阻塞。为了防止这种情况，内核会检查每个通过该操作发起的锁请求是否会导致死锁。若是，则内核会选择其中某个被阻塞的进程，让其fcntl()调用解除阻塞并出错 (EDEADLK)。对于Linux，最后调用fcntl()的进程会被选中，但SUSv3没有要求这种行为，并且有些UNIX实现也不会这样做。所有使用该操作的进程都必须准备处理错误EDEADLK。

　　即使对多个文件加锁，系统也能检测到死锁，涉及多个进程的循环死锁 (例如，进程A等待进程B持有的锁，进程B等待进程C持有的锁，而进程C等待进程A持有的锁) 同样如此。

55.2.7 锁限制和性能

　　SUSv3允许实现为可以获取的记录锁的数量设置固定的系统级上限。当达到该上限时，通过fcntl()加锁会导致错误ENOLCK。Linux没有为可以获取的记录数设置上限，所以可以获取的记录锁数仅受限于可用内存 (很多UNIX实现与之类似)。

　　用于实现记录锁的内核数据结构应该满足以下要求：

　　● 内核需要能够将新锁与同一进程持有的与新锁相邻的同模式的任何现有锁合并。

　　● 新锁可以完全替换调用进程持有的1个或多个锁。内核需要能够轻松找到所有这些锁。

　　● 当在现有锁的中间创建不同模式的新锁时，拆分现有锁的工作应该很简单。

　　内核为每个打开的文件维护了1个关联的锁链表。链表中的锁先按进程ID排序，再按起始偏移量排序。

　　内核还会在打开的文件关联的锁链表中维护flock()锁和文件租约。然而，这些类型的锁数量通常少得多，不太可能影响性能 (所以这里忽略了它们)。

　　每当向打开的文件关联的锁链表中添加新锁时，内核必须检查是否与现有锁冲突。这种搜索按顺序执行，从链表的头部开始。当大量锁分布在很多进程时，添加或删除锁所需的时间与文件上的锁的数量大致呈线性关系。

55.2.8 锁继承和释放的语义

　　fcntl()锁的继承和释放语义与flock()锁的继承和释放语义存在显著差异，以下几点值得注意：

　　● fork()创建的子进程不会继承父进程的记录锁。相比之下，子进程会继承同一文件锁的引用，并且可以通过该引用释放锁，进而导致父进程失去该锁。

　　● 记录锁会在exec()期间得到保留。

　　● 同一进程的所有线程共享记录锁。

　　● 记录锁同时与进程和i-node关联。因此，当进程终止时，其所有记录锁都会被释放。此外，当进程关闭文件描述符时 (显式调用close()、close-on-exec标志或调用dup2())，该进程持有的对应文件的锁都会被释放 (无论这些锁时通过哪个文件描述符获取的)。

　　记录锁的继承和释放语义是架构缺陷。它会导致库函数使用的记录锁出现问题，因为库函数无法阻止调用它的函数关闭引用库函数持有锁的文件的文件描述符 (释放锁)。替代实现方案是将锁与文件描述符关联。但是，当前的语义是记录锁的历史行为，也是现在的标准行为。这些语义极大地限制了记录锁的实用性。

55.3 强制锁

　　之前介绍的锁都是建议锁。这表示进程可以忽略fcntl() (或flock()) 的使用，直接对文件执行I/O。内核不会阻止这种行为。为了使用建议锁，应用程序设计者需要：

　　● 为文件设置合适的所有者 (或所属组) 和权限，以防止不相关进程对文件执行I/O。

　　● 确保应用程序的各个进程在执行I/O之前通过获取合适的锁来相互协作。

　　Linux和很多UNIX实现都允许fcntl()锁是强制锁。这表示所有文件I/O都会被检查，以确定其是否与其他进程在正在进行I/O的文件区域上持有的锁兼容。

55.3.1 强制锁对文件I/O的影响

　　当文件的强制锁已被启用，并且执行数据传输的系统调用遇到锁冲突时，若文件以阻塞模式被打开，则系统调用会被阻塞；否则，系统调用会立即导致错误EAGAIN (类似的规则也适用于truncate()和ftuncate())。

　　当文件以阻塞模式被打开时 (调用open()时指定标志O_NONBLOCK)，可能出现上图所示的死锁：进程A和B都以阻塞模式打开同一文件，获取不同文件部分的写入锁，然后写入对方锁定的区域。内核处理这种情况的方式与解决2个fcntl()调用之间的死锁的方式相同：选择其中的某个进程并让其write()导致错误EDEADLK。

　　当其他进程持有文件的任意部分的锁时，试图对该文件调用open()并指定标志O_TRUNC会导致错误EAGAIN。

　　当其他进程持有文件的任意部分的强制锁时，无法对该文件创建共享文件映射。相反，也无法对当前参与共享文件映射的文件的任意部分设置强制锁。对于这2种情况，相关系统调用都会导致错误EAGAIN (原因见49.3.2)。此外，这种文件I/O操作由内存子系统执行，而它并不知道系统中文件锁的位置。因此，为了防止映射更新存在强制锁的文件，内核会进行简单的检查——调用mmap()时判断文件中是否存在锁 (fcntl()则相反)。

55.3.2 强制锁注意事项

　　强制锁的作用比预期的更小，并且可能存在一些缺点和问题：

　　● 持有文件的强制锁不能防止其他进程删除该文件，因为取消链接文件只需要合适的父目录访问权限。

　　● 在启用公共可访问的文件的强制锁之前，应该仔细权衡利弊，因为特权进程也无法覆盖强制锁。恶意用户可以通过持续持有文件的强制锁来发起拒绝服务攻击 (虽然在大多数情况下，可以通过关闭set-group-ID位来恢复对文件的访问，但当强制文件锁导致系统被挂起时，这种方法就可能不适用了)。

　　● 强制锁会带来性能开销。对于每个以启用了强制锁的文件为目标的I/O系统调用，内核必须检查文件锁是否冲突。若文件上存在大量锁，则这种检查会显著降低I/O系统调用的速度。

　　● 强制锁会为应用程序设计带来额外开销，因为需要处理每个I/O系统调用可能导致错误EAGAIN或EDEADLK的情况。

　　● 由于当前Linux实现中的某些内核竞争条件，I/O系统调用在本应阻止它的强制锁存在时仍可能成功。

　　总之，最好避免使用强制锁。

55.4 文件/proc/locks

　　Linux特有的文件/proc/locks可用于获取系统中的锁的相关信息。

　　文件/proc/locks展示的信息包括fcntl()记录锁和flock()文件锁。每个锁显示的8个字段如下 (从左到右)：

　　● 第1个字段是该文件的所有锁中该锁的序号。

　　● 第2个字段是锁的类型 (POSIX表示记录锁，FLOCK表示文件锁)。

　　● 第3个字段是锁的模式 (建议锁或强制锁)。

　　● 第4个字是锁的类型 (读取锁或写入锁)。

　　● 第5个字是持有该锁的进程的进程ID。

　　● 第6个字段用于标识被锁定的文件。这3个由冒号分割的数字分别是文件所在文件系统的主设备号、次设备号和i-node号。

　　● 第7个字段是锁的起始字节 (文件锁的该字段的总是为0)。

　　● 第8个字段是锁的结束字节 (文件锁的该字段的总是为EOF)。

55.5 只运行程序的1个实例

　　有些程序——尤其是很多守护进程——需要确保系统上同时只有1个程序实例在运行。常见实现方法之一是让守护进程在标准目录下创建文件并获取该文件的写入锁。守护进程会在运行期间一直持有写入锁，并且在其终止前删除该文件。当守护进程的其他实例启动时，它将无法获取写入锁。因此，它会知道其他实例已在运行并终止 (很多网络服务器采用另一种约定：若服务器绑定的知名套接字端口已被占用，则默认该服务器的实例已在运行)。

　　守护进程的锁定的文件通常位于目录/var/run下。此外，可以通过在守护进程的配置文件中添加1行来指定该文件的位置。

55.6 早期上锁技术

　　缺乏文件锁定机制的早期UNIX实现采用了多种临时锁定技术。虽然这些技术已被fcntl()记录锁取代，但它们仍然出现在一些旧程序中。

　　SUSv3要求带有标志O_CREAT和O_EXCL的open()调用以原子方式检查文件是否存在并创建 (当多个进程都以这种方式打开文件时，只有1个进程会成功)。当与unlink()组合使用时，可以实现类似锁机制的效果。获取锁是通过成功使用这些标志打开创建文件并打开来实现。释放锁是通过unlink()来实现。虽然这种方法可行，但存在以下限制：

　　● open()出错表示其他进程持有该锁，所以必须循环调用open()，要么持续轮询 (浪费CPU时间)，要么在每次尝试之间设置延迟 (从锁可用到实际获取锁之间存在延迟)。

　　● 通过open()和unlink()获取和释放锁涉及文件操作，其速度远低于使用记录锁。

　　● 若进程在未删除锁文件的情况下意外终止，则锁不会被释放。对于这个问题，有一些临时解决方法 (包括检查文件的最后修改时间以及让持有锁的进程将其进程ID写入到文件以便检查该进程是否存在)，但这些方法都不是万无一失的。相比之下，记录锁会在进程终止时自动被释放。

　　● 当使用多个锁时，不会检测死锁。若发生死锁，则相关进程会一直被阻塞。相比之下，内核会检查记录锁的死锁。

　　● NFS第2版不支持O_EXCL语义。Linux 2.4 NFS也未能正确地实现该标志 (NFS第3版也是如此)。

　　link()会在新链接存在时出错的语义能用于实现锁定机制，同样使用unlink()来实现解锁机制。通常的做法是，让每个需要获取锁的进程生成唯一的临时文件名，通常包含进程ID (若锁文件被创建在网络文件系统上，则可能还会包含主机名)。为了获取锁，需要将该临时文件链接到某个约定的标准路径名 (硬链接的语义要求这些路径名必须位于同一文件系统)。
