《Linux/UNIX系统编程手册》第52章 POSIX消息队列

发表于 2026-05-26 更新于 2026-05-28 分类于 Linux 本文字数： 7.6k

　　POSIX消息队列与System V消息队列类似。但是，两者存在一些显著的差异：

　　● POSIX消息队列会被引用计数。被标记为待删除的消息队列只会在所有进程都将其关闭后才会被删除。

　　● System V消息有类型字段，并且通过msgrcv()接收信息时可以进行筛选。相比之下，POSIX消息有关联的优先级，并且总是会严格按优先级排序。

　　● POSIX消息队列允许在空队列中有消息可用时异步通知进程。

　　POSIX消息队列所需的支持在Linux 2.6.6中添加 (此外，还需要glibc 2.3.4或更高版本)。POSIX消息队列支持是可选内核组件，可以通过选项CONFIG_POSIX_MQUEUE配置。

　　POSIX消息队列API的主要函数如下：

　　● 函数mq_open()会创建消息队列或打开消息队列，并返回消息队列描述符供后续调用使用。

　　● 函数mq_send()会向队列写入1条消息。

　　● 函数mq_receive()会从队列读取1条消息。

　　● 函数mq_close()会关闭进程之前打开的消息队列。

　　● 函数mq_unlink()会删除消息队列名并将消息队列标记为待删除。

　　此外，POSIX消息队列API还包含以下特性：

　　● 每个消息队列有多个关联属性。部分属性会在通过mq_open()创建或打开队列时被设置。函数mq_getatrr()和mq_setattr()可用于检索和修改队列属性。

　　● 函数mq_notify()允许进程注册以接收来自队列的消息通知。当进程注册后，进程通过接收信号或由独立线程调用函数来获知消息可用。

　　消息队列描述符与打开的消息队列之间的关系类似于文件描述符与打开的文件之间的关系。消息队列描述符是进程级句柄，它引用系统级的打开的消息队列描述表中的某个条目，而该条目又引用某个消息队列对象。

　　对于Linux，POSIX消息队列被实现为虚拟文件系统中的i-node，而消息队列描述符和打开的消息队列描述分别被实现为文件描述符和打开的文件 (描述)。但是，SUSv3没有说明这一点，并且在某些UNIX实现中也不适用。

　　上图有助于阐明消息队列描述符的使用细节 (类似于文件描述符的使用)：

　　● 打开的消息队列描述有多个关联的标志。SUSv3仅详细说明了标志O_NONBLOCK (决定I/O是否以阻塞方式进行)。

　　● 多个进程可以持有引用同一消息队列描述的消息队列描述符 (进程打开消息队列并调用fork())。这些描述符共享标志O_NONBLOCK的状态。

　　● 多个进程可以持有引用不同消息队列描述的消息队列描述符，而这些消息队列描述又引用同一消息队列 (不同进程各自调用mq_open()打开消息队列)。

　　在任意时候，只能注册1个进程 (注册进程) 接收来自特定消息队列的通知。当某个消息队列已有对应的注册进程时，通过mq_notify()进一步注册进程会导致错误EBUSY。

　　当且仅当之前为空的消息队列有消息到达时，注册进程才会收到通知。若注册时队列中已有消息，则只有在队列清空后新消息到达时，注册进程才会收到通知。

　　当通知被发送给注册进程后，注册将会被删除。任何进程都可用重新注册以接收通知。换句话说，只要进程希望继续接收通知，就必须在每次收到通知后再次调用mq_notify()重新注册。

　　当且仅当没有其他进程因对队列调用mq_receive()而被阻塞时，注册进程才会收到通知；否则，被阻塞的进程会读取消息，注册进程仍然保持注册状态。

　　对于Linux，消息队列描述符实际上是文件描述符。可以通过I/O多路复用 (select()和poll()) 或epoll API来监控消息队列描述符。但是，该特性是非标准特性，并且SUSv3没有要求将消息队列描述符实现为文件描述符。

　　不同UNIX系统的POSIX消息队列的实现方式差异很大。某些UNIX系统使用用户空间实现，至少在1种这样的UNIX实现 (Solaris 10) 中，mq_open()手册页明确指出这种实现不安全。对于Linux，选择内核实现的原因之一是当时认为无法提供安全的用户空间实现。

52.1 mq_open()

　　函数mq_open()会创建或打开参数name标识的消息队列。若成功，则返回消息队列标识符 (类型为mqd_t)，后续调用可以使用该标识符来引用打开的消息队列。SUSv3对数据类型mqd_t唯一的规定是它不能是数组，即必须是可以在赋值语句中使用或作为函数参数按值转递的类型 (Linux将该数据类型实现为int，Solaris等UNIX实现将其实现为void *)。

　　参数oflag是控制mq_open()的各种行为的位掩码，可以包含以下标志：

oflag标志描述
O_CREAT消息队列不存在时创建
O_EXCL与O_CREAT一起使用时显式创建消息队列
O_RDONLY以只读模式打开
O_WRONLY以只写模式打开
O_RDWR以读写模式打开
O_NONBLOCK以非阻塞模式打开

　　oflag的用途之一是决定创建或打开消息队列。若该参数包含标志O_CREAT，则创建信息队列；否则，打开消息队列。oflag也能用于指定调用进程对消息队列的访问类型 (标志O_RDONLY、O_WRONLY或O_RDWR之一)。

　　标志O_NONBLOCK表示以非阻塞模式打开消息队列。后续对该队列的读取和写入 (mq_receive()和mq_send()) 不会阻塞进程，这些调用会出错 (EAGAIN)。

　　当使用mq_open()打开消息队列时，只需指定name和oflag。但是，若指定了标志O_CREAT，则还需要指定参数mode和attr (若name指定的队列存在，则这2个参数会被忽略)。

　　● mode是用于指定新消息队列的权限的位掩码。可以指定给该参数的标志与文件权限标志 (见15.4) 相同，并且该参数也受限于进程umask。为了读取消息队列 (mq_receive())，需要授予相应用户类别读取权限；为了写入消息队列 (mq_send())，需要授予相应用户类别写入权限。

　　● attr是用于指定新消息队列的属性的结构体mq_attr。若该参数为NULL，则使用实现定义的默认属性。

　　fork()创建的子进程会继承父进程的消息队列标识符，这些标识符仍然引用相同的打开的消息队列描述。子进程不会继承父进程的的消息通知注册。

52.2 mq_close()和mq_unlink()

　　函数mq_close()会关闭消息队列描述符mqdes。若调用进程已经通过mqdes注册来自该队列的消息通知，则消息注册会自动被删除，并且其他进程后续可以注册来自该队列的消息通知。

　　当进程执行exec()或终止时，其所有打开的消息队列描述符都会被关闭 (由于消息队列描述符的关闭，进程在对应队列上的消息通知注册也会被注销)。

　　与文件描述符相同，应该显式关闭不再需要的消息队列描述符，以防止进程耗尽消息队列描述符。与对文件描述符调用close()相同，关闭消息队列描述符不会删除它。

　　函数mq_unlink()会取消链接参数name指定的消息队列，并会将其标记为待删除 (一旦所有进程不再使用该消息队列，它就会被删除)。

52.3 消息队列属性

　　函数mq_open()、mq_getattr()和mq_setattr()都允许指定结构体mq_attr指针，头文件mqueue.h中定义了该结构体 (这3个函数只会使用该结构体的部分成员，在注释的括号中注明)：

　　该结构体的成员包含消息队列描述符对应的消息队列描述的相关信息 (mq_flags) 以及该描述符引用的消息队列的信息 (mq_maxmsg、mq_msgsize和mq_curmsgs)。

　　成员mq_maxmsg和mq_msgsize包含通过mq_open()创建消息队列时固定的信息，成员mq_flags和mq_curmsgs分别返回关于消息队列描述和消息队列当前状态的信息。

52.3.1 在创建消息队列时设置属性

　　当使用mq_open()创建消息队列时，结构体mq_attr的以下成员决定消息队列的属性：

　　● 成员mq_maxmsg定义了可以写入到队列的消息数上限，其值必须大于0。

　　● 成员mq_msgsize定义了可以写入到队列的消息的字节数上限，其值必须大于0。

　　这2个成员允许内核确定消息队列可能需要的最大内存占用。它们的值在创建消息队列时是固定的，后续无法进行修改。

52.3.2 mq_getattr()

　　函数mq_getattr()会返回参数mqdes关联的消息队列描述和消息队列的相关信息，结果存储在参数attr指向的结构体中。

　　除了成员mq_maxmsg和mq_msgsize之外，attr指向的结构体的成员mq_flags和mq_curmsgs也会被设置。

　　● 成员mq_flags是与mqdes关联的消息队列描述的标志，该成员仅包含标志O_NONBLOCK (该标志由mq_open()的参数oflag初始化，并且可以通过mq_setattr()修改)。

　　● 成员mq_curmsgs是队列中的当前消息数。当mq_getattr()返回时，若其他进程正在从队列读取消息或向其写入消息，则该信息可能已经发生变化。

52.3.3 mq_setattr()

　　函数mq_setattr()会设置参数mqdes关联的消息队列描述的属性，并可选择返回消息队列的相关信息。

　　● 该函数会使用参数newattr指向的结构体mq_attr中的成员mq_flags来修改mqdes关联的消息队列描述的标志。

　　● 若参数oldattr不为NULL，则该参数指向的结构体会包含之前的消息队列描述标志和消息队列属性。

　　SUSv3指出，唯一可以通过mq_setattr()修改的属性是标志O_NONBLOCK的状态。

　　考虑到特定实现可能定义了其他可修改的标志，或者SUSv3将来可能添加新标志，可移植应用程序应该先使用mq_getattr()获取mq_flags的值，然后修改标志O_NONBLOCK，最后调用mq_setattr()修改mq_flags (修改该标志的状态)。

52.4 mq_send()和mq_receive()

52.4.1 mq_send()

　　函数mq_send()会将消息msg_ptr添加到参数mqdes引用的消息队列。消息长度由参数msg_len指定，该参数必须不大于队列的属性mq_msgsize (否则会导致错误EMSGSIZE)。此外，消息长度可以为0。

　　每条消息有非负的整数优先级，由参数msg_prio指定。队列中的消息按优先级降序排列 (0的优先级最低)。当消息被添加到队列时，它会被放置在同一优先级的其他消息之后。当应用程序无需使用消息优先级时，可以始终将该参数指定为0。

　　SUSv3允许实现通过定义常量MQ_PRIO_MAX或sysconf(_SC_MQ_PRIO_MAX)调用的返回值来声明其消息优先级的上限。SUSv3要求该限制至少为32 (_POSIX_MQ_PRIO_MAX)，即优先级0~31可用。但是，不同实现的实际范围不同 (对于Linux，该常量为32768；对于Solaris，该常量为32；对于Tru64，该常量为256)。

　　若消息队列已满 (达到队列的属性mq_maxmsg)，则mq_send()会阻塞进程，直到队列中有足够空间；若指定了标志O_NONBLOCK，则立即会导致错误EAGAIN。

　　这个示例展示了函数mq_send()的命令行界面，其命令格式的详细信息显示在函数usageError()中。

52.4.2 mq_receive()

　　函数mq_receive()会从参数mqdes引用的消息队列中读取 (删除) 最高优先级的最旧信息，并将该消息返回到参数msg_ptr指向的缓冲区，该缓冲区的大小由参数msg_len指定。无论消息的实际大小如何，msg_len都必须不小于队列的属性mq_msgsize (否则，mq_receive()会导致错误EMSGSIZE)。

　　若参数msg_prio不为NULL，则接收的消息的优先级会被复制到该参数指向的缓冲区。

　　若消息队列为空，则mq_receive()会阻塞进程，直到队列中有消息可用。若指定了标志O_NONBLOCK，则立即会导致错误EAGAIN。

52.4.3 mq_timedsend()和mq_timedreceive()

　　函数mq_timedsend()和mq_timedreceive()与mq_send()和mq_receive()完全相同。若操作不能立即完成，并且未启用消息队列描述的标志O_NONBLOCK，则参数abs_timeout指定调用阻塞的时间限制。这2个函数超时都会导致错误ETIMEDOUT。

　　abs_timeout通过自纪元以来的以秒和纳秒为单位的绝对值来指定超时时间。为了设置相对超时时间，可以通过clock_gettime()获取CLOCK_REALTIME时钟的当前值并加上所需的值来获取合适的结构体timespec。

　　对于Linux，将abs_timeout指定为NULL表示无超时时间，但SUSv3没有说明这种行为。因此，可移植应用程序不应该依赖于该行为。

　　mq_timedsend()和mq_timedreceive()源于POSIX.1b (1999)，但并非所有UNIX实现都提供了这2个函数。

52.5 mq_notify()

　　POSIX消息队列和System V消息队列的区别之一是能够在之前为空的队列上收到消息可用的异步通知 (即消息队列非空)。进程可以选择信号或在单独线程中调用函数来接收通知。

　　POSIX消息队列的通知特性与POSIX间隔定时器 (见23.6) 的通知机制类似，这2个API都源于POSIX.1b。

　　函数mq_notify()会注册调用进程，以便在参数mqdes指定的空消息队列有消息可用时收到通知。

　　参数notification指定进程收到通知的方式。注册进程可以通过调用mq_notify()并将该参数指定为NULL来显式注销自己作为消息通知的目标。

　　在23.6.1中介绍了用于指定notification类型的结构体sigevent。这里仅展示该结构体的与mq_notify()相关的成员：

　　成员sigev_notify可以被指定为以下值之一：

　　● SIGEV_NONE表示注册进程以接收通知，但当之前为空的队列有消息到达时，该进程不会收到通知，并且会被注销。

　　● SIGEV_SIGNAL表示通过生成信号sigev_signo来通知进程。若该信号为实时信号，则成员sigev_value可用于指定随附数据。可以通过传递给信号处理函数的结构体siginfo_t的成员si_value或通过调用sigwaitinfo()或sigtimedwait()返回的数据来获取随附数据。在这种情况下，结构体siginfo_t的其他成员也会被赋值，si_code会被设置为SI_MESGQ，si_signo会被设置为信号编号，si_pid和si_uid会被设置为发送消息的进程的进程ID和实际用户ID (大多数UNIX实现不会设置这2个成员)。

　　● SIGEV_THREAD表示调用成员sigev_notify_function指向的函数来通知进程，就像它是新线程的启动函数一样。该成员可以被指定为NULL或结构体pthread_attr_t指针。成员sigev_value会作为参数被传递给该函数。

52.5.1 通过信号接收消息通知

　　这个示例展示了如何通过信号接收消息通知，该程序会执行以下步骤：

　　1) 以非阻塞方式打开命令行参数指定的命名消息队列，设置队列的属性mq_msgsize，并为消息分配指定大小的缓冲区 (第28~43行)。

　　2) 阻塞通知信号 (SIGUSR1) 并为其注册处理函数 (第45~58行)。

　　3) 调用mq_notify()来注册进程 (第60~65行)。

　　4) 无限循环来接收消息。每次循环会调用sigsuspend()和mq_notify()，然后调用mq_receive()来接收信息 (第71~86行)。

　　关于该程序，以下几点值得注意：

　　● 这里使用sigsuspend()阻塞并等待通知信号 (而非pause())，以防在其他地方执行时在for循环中可能错过信号的情况。若出现这种情况，并且使用pause()等待信号，则下一次调用pause()会被阻塞 (即使信号已到达)。

　　● 队列以非阻塞方式被打开，所以无论通知何时到达，用while循环都会读取队列中的所有消息 (这会确保新消息到达时会生成消息通知)。此外，使用非阻塞方式表示消息队列为空时mq_receive()会导致错误EAGAIN。

　　● 对于for循环，在读取队列中的所有消息之前，重新注册进程很重要。若以相反的顺序的执行这2步，则可能出现情况：从队列中读取所有消息后while循环终止，然后消息被写入到队列，最后调用mq_notify()重新注册进程。此时，不会生成消息通知 (因为队列不为空)，而运行程序的进程会被阻塞在下一次调用sigsuspend()。

52.5.2 通过线程接收消息

　　这个示例展示了如何通过线程接收消息通知。该程序采用了与mq_notify_sig.c类似的设计。

　　● 当收到消息通知时，该程序会在读取队列前重新注册进程 (第29~39行)。

　　● 该程序也会使用非阻塞模式打开消息队列 (第69~73行)，所以收到通知后可以读取队列中的所有消息。

　　关于该程序，以下几点值得注意：

　　● 该程序通过将传递给mq_notify()的结构体sigevent的成员的sigev_notify指定为SIGEV_THREAD来将接收消息通知的方式指定为线程。线程启动函数threadFunc()由成员sigev_notify_function()指定。

　　● 当启用消息通知后，主线程会无期限地暂停。定时器通知会通过调用独立线程中的threadFunc()来发送。

　　● 实际上，可以将消息队列描述符mqd声明为全局变量。但是，这里使用的是另一种方式：将消息队列描述符作为参数传递给线程启动函数。这里必须将消息队列描述符指针赋值给sev.sigev_value.sival_ptr，因为除了要求mqd_t不为数组之外，SUSv3没有规定该数据类型的性质或大小。

52.7 消息队列限制

　　SUSv3定义了2个POSIX消息队列限制：MQ_PRIO_MAX和MQ_OPEN_MAX。

　　● 限制MQ_PRIO_MAX定义了消息的最大优先级。

　　● 实现可以定义限制MQ_OPEN_MAX来表示进程能够持有的打开状态的消息队列数的上限。SUSv3要求该限制至少为_POSIX_MQ_OPEN_MAX (8)。Linux没有定义该限制。相反，由于Linux将消息队列描述符实现为文件描述符，所以适用的限制是文件描述符资源限制 (见36.3)。

　　除了这2个限制之外，Linux还提供了一些/proc文件，用于查看和修改POSIX消息队列使用的限制 (修改需要特权)。以下3个文件位于目录/proc/sys/fs/mqueue下：

　　● 文件msg_max定义了通过mq_open()创建的消息队列的属性mq_maxmsg的上限。该限制默认为10，最小值为1 (Linux 2.6.28之前是10)，最大值由内核常量HARD_MSGMAX定义 (该常量的值为131072 / sizeof(void *))。当特权进程 (能力CAP_SYS_RESOURCE) 调用mq_open()时，该文件指定的限制会被忽略，HARD_MSGMAX将作为mq_maxmsg的上限值。

　　● 文件msgsize_max定义了非特权进程通过mq_open()创建的消息队列的属性mq_msgsize的上限。该限制默认为8192，最小值为128 (Linux 2.6.28之前是8192)，最大值为220 (Linux 2.6.28之前是INT_MAX)。该限制不会影响特权进程 (能力CAP_SYS_RESOURCE) 调用mq_open()创建的消息队列。

　　● 文件queues_max是可以创建的消息队列数的系统级限制。一旦达到该限制，继续创建消息队列需要特权 (能力CAP_SYS_RESOURCE)。该限制默认为256，可以被修改为0到INT_MAX之间的任意值。

　　Linux的资源限制RLIMIT_MSGQUEUE可用于限制调用进程的实际用户ID的所有消息队列可以占用的空间大小。
