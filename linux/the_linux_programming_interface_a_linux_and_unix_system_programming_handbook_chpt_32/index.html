《Linux/UNIX系统编程手册》第32章 线程：线程取消

发表于 2025-08-13 更新于 2025-08-14 分类于 Linux 本文字数： 7k

　　取消线程是向线程发送请求，要求它立即终止。

　　函数pthread_cancel()会发送取消请求给参数thread指定的线程。一旦请求发送完成，该函数会立即返回，即不会等待目标线程终止。目标线程如何处理取消请求取决于其取消状态和类型。

　　函数pthread_setcancelstate()会将调用线程的取消状态设置为参数state指定的状态，该参数可以被指定为PTHREAD_CANCEL_DISABLE和PTHREAD_CANCEL_ENABLE。

　　● PTHREAD_CANCEL_DISABLE表示线程不可取消。若处于该状态的线程收到取消请求，取消请求将保持待处理状态，直到线程退出不取消状态。

　　● PTHREAD_CANCEL_ENABLE表示线程可取消，这也是新线程的默认取消状态。

　　调用线程的旧取消状态会被存储在参数oldstate指向的缓冲区中。Linux和很多UNIX实现都允许将该参数指定为NULL。但是，SUSv3未说明该特性，所以可移植应用程序不应该将该参数指定为NULL。

　　函数pthread_setcanceltype()会将调用线程的取消类型设置为参数type指定的类型，该参数可以被指定为PTHREAD_CANCEL_ASYNCHRONOUS和PTHREAD_CANCEL_DEFERRED。

　　● PTHREAD_CANCEL_ASYNCHRONOUS表示线程可能随时会被取消。

　　● PTHREAD_CANCEL_DEFERRED表示取消实际会发生在取消点，这也是新线程的默认取消类型。

　　调用线程的旧取消类型会被存储在参数oldtype指向的缓冲区中。Linux和很多UNIX实现都允许将该参数指定为NULL。但是，SUSv3未说明该特性，所以可移植应用程序不应该将该参数指定为NULL。

　　当线程调用fork()后，子进程会继承调用线程的取消类型和状态。当线程调用exec()后，新程序的主线程的取消状态和类型会被分别设置为PTHREAD_CANCEL_ENABLE和PTHREAD_CANCEL_DEFERRED。

32.1 取消点

　　若线程的取消状态和类型分别为PTHREAD_CANCEL_ENABLE和PTHREAD_CANCEL_DEFERRED，则取消请求会在线程下一次到达取消点时被执行。若该线程不是分离线程，则进程中的其他线程必须合并该线程 (防止它转换为僵尸线程)。当合并被取消的线程时，pthread_join()的参数retval会返回特殊值PTHREAD_CANCELED。

　　取消点是对实现定义的诸多函数之一的调用。SUSv3要求，若实现提供了以下函数，则这些函数必须是取消点 (其中的大部分函数都能够无期限地阻塞线程)：

　　SUSv3还定义了大量可以是取消点的函数，包括stdio函数、dlopen API、syslog API、nftw()、popen()、semop()、unlink()以及从utmp等系统文件检索信息的各种函数。可移植应用程序必须正确地处理在调用这些函数时线程可能被取消的情况。

　　SUSv3要求，除了必须和可以作为取消点的函数之外，标准中的任何其他函数都不能作为取消点 (所以可移植应用程序不需要处理这些其他函数可能导致线程取消的情况)。

　　SUSv4将openat()添加到必须是取消点的函数列表，并从该列表中删除了sigpause() (移动到了可以是取消点的函数列表) 和usleep() (从标准中删除)。

　　实现可以随意地将标准未说明的函数标记为取消点，即任何可能会阻塞的非标准函数 (可能是因为它可能会访问文件) 都可能是取消点。因此，glibc中的很多非标准函数都被标记为取消点。

　　这个示例展示了pthread_cancel()的用法。主线程会创建1个执行无限循环的线程，新线程每次循环会睡眠1秒并打印循环计数器的值 (当向该线程发送取消请求或进程终止时，该线程才会终止)。同时，主线程会睡眠3秒，然后向新线程发送取消请求。

　　对于这个示例，新线程会接收取消请求，因为该线程执行的函数中有1个取消点 (sleep())。但是，假设线程执行的循环没有取消点 (例如，计算密集型循环)。在这种情况下，线程永远不会接收取消请求。

　　函数pthread_testcancel()的功能是充当取消点。当线程执行的代码中没有取消点时，可以定期调用该函数来确保该线程能够及时响应其他线程发送的取消请求。

32.2 清除处理函数

　　当有待处理取消请求的线程到达取消点并直接终止时，共享变量和Pthreads对象 (例如，互斥锁) 可能会处于不一致状态，这可能会导致进程中的其他线程产生错误结果、死锁或崩溃。为了解决该问题，线程可以注册清除处理函数。清除处理函数在线程终止前可以执行修改全局变量和解锁互斥锁等任务。

　　线程可以有多个清除处理函数。当线程被取消时，清除处理函数会自动被调用，调用顺序与栈结构类似，即最近注册的清除处理函数最先被调用。当所有清除处理函数都被调用后，线程会终止。

　　函数pthread_cleanup_push()会将参数routine指向的函数添加到调用线程的清除处理函数栈的栈顶，而参数arg将作为参数传递给该清除处理函数 (尽管该参数的数据类型是void *，但通过合适类型转，该参数也能传递标量数据)。

　　函数pthread_cleanup_pop()会删除调用线程的清除处理函数栈的栈顶清除处理函数。若参数execute不为0，则被删除的清除处理函数还是会被调用。

　　通常，只有在执行特点代码段期间取消线程时才需要执行清除操作。若线程到达该代码段末尾时未被取消，则不需要清理操作。因此，pthread_cleanup_push()调用应该与pthread_cleanup_pop()调用成对出现。

　　虽然这里将pthread_cleanup_push()和pthread_cleanup_pop()描述为函数，但SUSv3允许它们被实现为宏，分别扩展为包括左括号{和右括号}的语句序列。并非所有UNIX实现都这样做，但Linux和很多系统都这样做。这表示每个pthread_cleanup_push()调用都必须在同一词法块中有1个对应的pthread_cleanup_pop()调用 (对于这种实现方式，pthread_cleanup_push()和pthread_cleanup_pop()之间的声明的变量将被限制在该词法范围内)。

　　为了便利，若线程通过pthread_exit()终止，则任何尚未弹出的清除处理函数也会自动被调用。

　　这个示例展示了清除处理函数的用法。主线程会创建1个线程 (第64~68行)，新线程会先分配内存 (第33行)，再锁定互斥锁mtx (第36~40行)。因为新线程可能会被取消，所以这里通过pthread_cleanup_push()注册清除处理函数，并将buf作为参数传递。当清除处理函数被调用时，会释放内存和解锁互斥锁 (第14~26行)。最后，新线程会循环等待条件变量cond (第44~51行)。该循环会以2种方式之一终止，这取决于是否提供了命令行参数：

　　● 若未提供命令行参数，则新线程会被主线程取消 (第75~80行)。在这种情况下，新线程的取消实际发生在调用pthread_cond_wait()。作为取消的一部分，通过pthread_cleanup_push()注册的清除处理函数会自动被调用。

　　● 若提供了命令行参数，则主线程会将glob设置为1并执行唤醒操作 (第85~91行)。在这种情况下，新线程会一直执行到pthread_cleanup_pop(1) (第54行)，这会调用清除处理函数。

　　最后，主线程会合并新线程，并打印新线程是被取消还是正常终止 (第94~106行)。

32.3 异步取消

　　若线程的取消类型是PTHREAD_CANCEL_ASYNCHRONOUS，则该线程可能随时被取消，取消请求的传递不会被推迟到线程下一次到达取消点。

　　异步取消的问题在于，尽管清除处理函数仍然会被调用，但它无法获取线程的状态。对于32.2中的示例程序中的新线程，取消类型是PTHREAD_CANCEL_DEFERRED，取消点只有pthread_cond_wait()。到那时，buf已被初始化为指向已分配的内存块，互斥锁mtx已被锁定。但是，由于异步取消，线程可能随时被取消 (例如，调用malloc()之前，在调用malloc()和锁定互斥锁之间或锁定互斥锁之后)。清除处理函数无法得知取消发生在何处或究竟需要执行哪些清理步骤。此外，取消可能发生在malloc()调用期间，这很可能会导致混乱的情况。

　　通常，异步取消线程不能分配任何资源以及获取互斥锁、信号量或锁。这表示无法使用很多库函数，包括大多数Pthreads函数 (SUSv3对pthread_cancel()、pthread_setcancelstate()和pthread_setcanceltype()做出了例外，明确要求将它们实现为异步取消安全的，即能够在异步取消线程中安全被调用)。因此，异步取消有用的情况很少 (例如，取消处于计算密集型循环中的线程)。
