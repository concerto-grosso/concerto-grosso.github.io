《Linux/UNIX系统编程手册》第24章 进程创建

发表于 2025-05-22 更新于 2025-05-23 分类于 Linux 本文字数： 8.5k

　　对于很多应用程序，创建多个进程是划分任务的有效方式之一。例如，网络服务器进程可能会监听客户端请求，然后创建子进程来处理器每个请求。同时，服务器进程继续监听。

24.1 fork()

　　系统调用fork()会创建1个子进程，子进程与父进程 (调用进程) 大致相同。这2个进程都会从fork()返回的位置继续执行。该系统调用的名称源于事实：可以将父进程视为分裂为2个自己的副本。

　　父进程和子进程执行相同的程序，但它们有独立的栈、堆和数据段副本。子进程的栈、堆和数据段最初与父进程内存中的对应部分相同。在fork()返回后，父进程和子进程可以修改自己的栈、堆和数据段中的变量 (不会相互影响)。

　　对于父进程，fork()会返回子进程的进程ID。对于子进程，fork()会返回0。若无法创建子进程，则fork()会返回-1，这可能因为超过该实际用户ID允许的进程数量资源限制 (RLIMIT_NPROC) 或达到可以创建的进程数量的系统级上限。

24.1.1 父进程和子进程之间的文件共享

　　当fork()成功时，子进程会继承父进程的所有文件描述符。继承文件描述符的方式如同dup() (见5.3)，这表示父进程和子进程的文件描述符引用的是同一打开的文件句柄。因为打开的文件句柄包含文件偏移量、打开文件状态标志，所以父进程和子进程共享打开的文件的这些属性。

　　这个示例会通过mkstemp()打开1个临时文件，然后调用fork()创建子进程。子进程会修改临时文件的文件偏移量和打开文件状态标志。最后，父进程会检索文件偏移量和打开文件状态标志。

24.1.2 fork()的内存语义

　　从概念角度，fork()可是视为创建父进程的栈、堆、文本段和数据段的副本 (对于某些早期UNIX实现，确实如此)。但是，直接将父进程的虚拟内存页复制给子进程很浪费。fork()后紧接的通常是exec()，这会将进程的文本段替换为新程序并重新初始化进程的栈、堆和数据段。Linux和大多数现代UNIX实现使用了2种技术来避免这种浪费：

　　● 内核将父进程和子进程的文本段标记为只读，这表示父进程和子进程共享同一文本段。fork()通过为子进程构建一些页表条目来为创建子进程的文本段，这些页表条目引用父进程使用的虚拟内存页。

　　● 对于父进程的栈、堆和数据段，内核会使用写时复制技术。内核先设置子进程的这些段的页表条目，以便它们与父进程的对应页表条目引用同一物理内存页 (这些物理页为只读)。在调用fork()后，父进程或子进程试图修改这些页时会触发故障，内核会复制待修改的页，将其分配给导致故障的进程并适当地修改相应的页表条目。从此以后，父进程和子进程可以各自修改这些页的私有副本 (做出的修改对彼此不可见)。

21.1.4 竞争条件

　　在调用fork()后，父进程和子进程的调度顺序是不确定的 (对于多处理器系统，父进程和子进程可能同时调度)。对于 (显式或隐式) 依赖执行顺序的应用程序，可能由于竞争条件而导致意料之外的结果。

　　这个示例展示了这种不确定性。这个程序会循环调用fork()来创建子进程，然后打印包含循环次数和指示进程是父进程还是子进程的字符串。

　　对于Linux 2.6.32及以上版本，默认情况是fork()后父进程先于子进程被调度。这可以通过将特定于Linux的文件/proc/sys/kernel/sched_child_runs_first赋值为非0值来修改。

　　因此，在开发程序时不应该假设fork()后父进程和子进程的调度顺序。为了确保父进程和子进程的调度顺序，需要使用同步机制 (例如，信号、信号量、文件锁以及消息传递)。

24.1.5 通过信号同步避免竞争条件

　　这个示例展示了如何通过信号同步避免竞争条件。这里假设父进程必须等待子进程执行某些操作 (若子进程必须等待父进程执行某些操作，则可以将信号相关的函数调用互换。有时，父进程和子进程会相互发送多次信号来协调行为，但这种协调通常都是通过信号量、文件锁或消息传递来完成)。在调用fork()前需要先阻塞同步信号。当父进程在调用fork()后需要再次阻塞同步信号时，它还是容易受到竞争条件的影响 (这里假设子进程的信号掩码无关紧要，若有必要，则可以像父进程一样可以恢复信号掩码)。

24.2 vfork()

　　早期BSD实现的fork()会直接复制父进程的栈、堆和数据段，但这种行为很浪费。因此，后续版本的BSD添加了系统调用vfork()。vfork()比fork()更高效，但行为有所不同。

　　现代UNIX实现使用写时复制技术实现fork()，这大幅提升了其效率，所以在很大程度上不再需要vfork()。尽管如此，Linux和很多UNIX实现还是提供了BSD语义的vfork()。但是，vfork()不同寻常的语义可能会导致细微的程序漏洞，所以通常不会使用该系统调用 (除非极其需要vfork()带来的性能提升)。

　　与fork()类似，vfork()也会创建1个子进程。但是，vfork()旨在用于那些子进程立即调用exec()的程序中。

　　vfork()与fork()的2个不同之处使vfork()更高效：

　　● vfork()不会为子进程复制虚拟内存页或页表。相反，子进程会使用父进程的内存，直到子进程成功调用exec()或通过_exit()终止。

　　● vfork()会导致父进程的执行会被挂起，直到子进程成功调用exec()或通过_exit()终止。

　　这2个不同之处会产生一些影响。由于子进程会使用父进程的内存，一旦父进程恢复，子进程对栈、堆和数据段的修改都会被父进程看到。此外，当子进程在vfork()和后续的exec()或_exit()之间执行函数返回时，也会影响父进程。这与6.6.3中的通过longjmp()非本地跳转到已经返回的函数类似，并且通常会导致段故障 (SIGSEGV)。

　　子进程在vfork()与exec()之间能执行的不影响父进程的操作很少，其中包括对打开的文件描述符的操作 (但不包括stdio文件流)，因为每个进程的文件描述符表维护在内核空间，并且在执行fork()时会被复制。

　　vfork()的语义会保证子进程先于父进程调度，并且父进程会一直阻塞，直到子进程成功调用exec()或通过_exit()终止。

　　当使用vfork()时，尽量保证子进程立即调用exec()。若exec()出错，则子进程应该调用_exit()终止。

　　SUSv3规定，以下3种情况会导致程序的行为是未定义的：

　　● 修改用于存储vfork()的返回值的变量以外的数据。

　　● 从调用vfork()的函数中返回。

　　● 在成功调用exec()或通过_exit()终止之前调用其他函数。

　　SUSv3将vfork()标记为过时，SUSv4删除了vfork()的规范。SUSv3没有详细说明vfork()的很多细节，并且允许将其实现为调用fork()。当vfork()以这种方式实现时，BSD语义将不再生效 (内核2.0及以下版本的Linux和有些UNIX实现就是如此)。
