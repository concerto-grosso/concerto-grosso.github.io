《Linux/UNIX系统编程手册》第25章 进程终止

发表于 2025-06-06 更新于 2025-06-07 分类于 Linux 本文字数： 5.2k

　　进程可以通过2种方式终止。第1种方式是正常终止 (通过系统调用_exit()或从主函数返回)。第2种方式是异常终止，因为收到默认处理方式是终止进程的信号 (无论是否生成核心转储文件)。

　　对于系统调用_exit()，参数status用于指定进程的终止状态。父进程可以调用wait()来获取子进程的终止状态。尽管该参数的数据类型为int，但父进程只能访问低8位。该系统调用总是会终止进程 (即不会返回)。

　　按照惯例，终止状态为0时表示进程成功终止，终止状态不为0时表示进程没有成功终止。如何解释终止状态不为0的情况因应用程序而异。SUSv3详细说明了2个常量：EXIT_SUCCESS (0) 和EXIT_FAILURE (1)。

　　程序通常不会直接调用_exit()，而是调用库函数exit()。

　　库函数exit()会执行以下操作：

　　● 按照退出处理函数注册顺序的逆序调用退出处理函数。

　　● 刷新stdio流缓冲区。

　　● 以参数status指定的值调用_exit()。

　　不同于特定于UNIX的_exit()，exit()是C标准库的一部分，所有C实现都提供了该函数。

　　在主函数中调用return n通常等同于调用exit(n)，因为调用主函数的运行时函数在调用exit()时使用的是主函数的返回值。

　　若退出处理过程包括访问主函数中的局部变量 (例如，调用setvbuf()和setbuf()时指定了主函数中的局部变量)，则从主函数返回将会导致未定义的结果。在这种情况下，调用exit()不等同于从主函数返回。

　　从主函数返回时不指定返回值或在主函数的末尾退出也会导致调用的主函数的函数调用exit()，但具体结果取决于使用的C标准版本和编译选项：

　　● 对于C89，这些情况下的行为是未定义的，程序可能以任意status值返回。这也是Linux上的gcc的默认行为，程序的终止状态将会是栈中的某个随机值或取自特定CPU寄存器，所以应该避免通过这种方式终止程序。

　　● 对于C99，在主函数的末尾退出等同于调用exit(0)。这也是在Linux上使用gcc –std=c99编译程序时的行为。

25.1 进程终止细节

　　当进程终止时，会执行以下操作：

　　● 打开的文件描述符、目录流、消息目录描述符和转换描述符都会被关闭。

　　● 因为关闭了文件描述符，所以进程持有的所有文件锁都会被释放。

　　● 相关的System V共享内存段会被分离，并且这些内存段对应的shm_nattch计数器会减1。

　　● 对于所有semadj值由进程设置的System V信号量，其值会增加semadj。

　　● 若进程是控制终端的控制进程，则控制终端的前台进程组的所有进程都会收到信号SIGHUP，然后终端会与会话取消关联。

　　● 打开的POSIX命名信号量会被关闭 (如同调用sem_close())。

　　● 打开的POSIX消息队列会被关闭 (如同调用mq_close())。

　　● 进程的终止会导致某个进程组变成孤儿进程组。若该进程组中存在停止的进程，则该进程组的所有进程都会先收到信号SIGHUP，再收到信号SIGCONT。

　　● 进程通过mlock()或mlockall()设置的内存锁会被删除。

　　● 进程通过mmap()设置的内存映射会被解除映射。

25.2 退出处理函数

　　有时，应用程序在进程终止时会自动执行一些操作。以应用程序库为例，应用程序库会在进程的生命周期内使用，并且在进程退出时需要自动执行一些清理操作。应用程序库不能控制进程的生命周期，并且无法强制主函数在退出前调用特定的清理函数，所以清理操作可能不会执行。解决方法之一是使用退出处理函数 (较旧的System V手册使用的是术语程序终止例程)。

　　退出处理函数是程序员提供的函数，该函数会在进程的生命周期的某个时间点注册。当进程通过exit()正常终止时，会自动调用退出处理函数。若程序直接调用_exit()或者进程被信号终止，则不会自动调用退出处理函数。

　　在某种程度上，进程因信号终止时不自动调用退出处理函数限制了退出处理函数的实用性。对于这种情况，最好的处理方法是注册信号处理函数，并在信号处理函数中设置会导致主函数调用exit()的全局标志 (因为exit()是非异步信号安全函数，所以不能直接在信号处理函数中直接调用exit())。但是，这也无法处理信号SIGKILL，因为该信号的处理方式无法被修改，这也是避免使用该信号来终止进程的理由之一。

25.2.1 注册退出处理函数

　　glibc提供了2种注册退出处理函数的方法。第1种方法是使用函数atexit()，SUSv3详细说明了这种方法。

　　函数atexit()会将参数func指向的函数添加1个函数列表，调用进程在终止会调用该函数列表中的所有函数。func指向的函数应该声明为没有参数和返回值：

　　atexit()在出错时返回的是非0值，而非-1。

　　退出处理函数可以注册多个 (同一退出处理函数甚至可以注册多次)。当程序调用exit()时，这些退出处理函数会按照注册顺序的逆序被调用。

　　从理论角度，退出处理函数中可以执行任意操作 (包括注册退出处理函数)。但是，若某个退出处理函数因调用_exit()或进程被信号终止 (例如，退出处理函数调用raise()) 而无法返回，则剩余的退出处理函数不会被调用。此外，exit()通常会执行的其他操作 (即刷新stdio缓冲区) 也不会被执行。

　　SUSv3规定，若退出处理函数调用exit()，则结果将是未定义的。对于Linux，剩余的退出处理函数会正常被调用。但是，对于有些系统，这会导致再次调用所有退出处理函数，进而导致无限递归 (直到栈溢出终止进程)。

　　SUSv3要求实现允许单个进程能够注册至少32个退出处理函数。可以通过调用sysconf(_SC_ATEXIT_MAX)来获取实现定义的可以注册的退出处理函数的数量上限 (但是，无法获取已注册的退出处理函数的数量)。通过将已注册的退出处理器函数链接在动态分配的链表中，glibc允许注册几乎无限数量的退出处理函数。对于Linux，可以注册的退出处理函数的数量上限是2147482647 (即32位有符号整数的最大值)。

　　通过fork()创建的子进程会继承父进程注册的退出处理函数。当进程调用exec()时，所有注册的退出处理函数会被删除 (因为exec()会替换退出处理函数的代码和现有程序的剩余代码)。

　　通过atexit()注册的退出处理函数存在很多限制。第1个限制是调用退出处理函数时退出处理函数无法得知传递给exit()的终止状态。第2个限制是退出处理函数没有参数。为此，glibc提供了非标准函数on_exit()。

　　参数status是传递给exit()的终止状态。参数arg可以是指向结构体的指针或标量类型。

　　与atexit()相同，on_exit()在出错时会返回非0值。同一进程可以通过on_exit()注册多个退出处理函数，并且通过atexit()和on_exit()注册的退出处理函数位于同一函数列表 (若程序中同时使用了这2个函数，则退出处理函数会按照通过这2个函数注册的顺序的逆序被调用)。

　　stdio缓冲区维护在进程的用户空间内存 (见13.2)，所以子进程会继承父进程的stdio缓冲区。将标准输出重定向到终端时默认使用的是行缓冲，所以通过printf()写入的以换行符结尾的字符串会立即出现在终端。但是，将标准输出重定向到文件时默认使用的是块缓冲，所以调用fork()时通过printf()写入的字符串仍然位于父进程的stdio缓冲区中，子进程会继承该缓冲区。当随后调用exit()时，会刷新父进程和子进程的stdio缓冲区，导致通过printf()写入的字符串重复出现。

　　可以通过以下方法之一避免这种情况：

　　● 对于stdio缓冲区，可以在调用fork()之前先通过fflush()来刷新stdio缓冲区，也可以通过setvbuf()或setbuf()来禁用stdio流的缓冲。

　　● 在子进程中调用_exit()来防止刷新stdio缓冲区。这种方法体现了1个更通用的原则：对于多进程应用程序，通常只有1个进程 (大多数情况下是父进程) 使用exit()终止，其他进程使用_exit()终止。这会保证只有1个进程会调用退出处理函数和刷新stdio缓冲区。

　　这个示例中通过write()写入的字符串不会重复出现，因为write()会直接将数据传输到内核缓冲区，并且内核缓冲区不会被fork()复制。stdio缓冲区中的数据只会在stdio缓冲区被刷新时才会被传输，所以通过write()写入的字符串出现在通过printf()写入的字符串之前。
