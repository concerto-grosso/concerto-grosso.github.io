《Linux/UNIX系统编程手册》第54章 POSIX共享内存

发表于 2026-06-06 分类于 Linux 本文字数： 3k

　　System V共享内存和共享文件映射都有潜在的缺点：

　　● 使用键和标识符的System V共享内存与使用文件名和描述符的标准UNIX I/O模型不一致。这表示需要一整套全新的系统调用和命令来操作System V共享内存段。

　　● 将共享文件映射用于IPC需要创建磁盘文件 (即使无需为共享区域设置持久化存储)。此外，这种技术会产生文件I/O开销。

　　正是因为这些缺点，POSIX.1b定义了新的共享内存API：POSIX共享内存 (POSIX中使用的术语是共享内存对象，System V中使用的术语是共享内存段。这一差异源于历史原因，但它们的含义相同)。

　　POSIX共享内存允许不相关进程在不创建映射文件的情况下共享内存区域。Linux从2.4开始支持POSIX共享内存。

　　SUSv3要求POSIX共享内存对象至少具有内核持久性，但没有说明POSIX共享内存的任何实现细节，尤其是没有要求使用文件系统来标识共享内存对象 (但很多UNIX实现都这样做)。有些UNIX实现将共享内存的名称创建为标准文件系统中特殊位置的文件。Linux使用的是挂载在目录/dev/shm下的具有内核持久性的tmpfs文件系统。

　　系统中的所有POSIX共享内存区域的总大小受限于tmpfs文件系统的大小。该文件系统通常在系统引导时以某个默认大小 (例如，256M) 被挂载。若有必要，超级用户可以通过命令mount -o remount,size=<num-bytes>重新挂载该文件系统来修改其大小。

　　为了使用POSIX共享内存对象，需要执行以下步骤：

　　1) 使用函数shm_open()以指定名称打开对象。该函数与open()类似，它会创建或打开共享内存对象，并返回引用该对象的文件描述符。

　　2) 将第1步获取的文件描述符传递给mmap()并在参数flags中指定MAP_SHARED。这会将共享内存对象映射到进程虚拟地址空间。与mmap()的其他用法相同，一旦映射完成，就可以关闭文件描述符 (而不影响映射)，或者保留该文件描述符供后续的fstat()和ftruncate()调用使用。

　　对于POSIX共享内存，shm_open()和mmap()之间的关系类似于System V共享内存中shmget()和shmat()之间的关系。使用POSIX共享内存时，之所以使用上述步骤，而非使用1个能同时完成这2步的函数，是有历史原因的。当POSIX委员会添加该特性时，mmap()就已经存在。

　　System V共享内存、共享文件映射和POSIX共享内存具有以下相同点：

　　● 它们都提供了快速IPC，并且应用程序通常需要信号量等同步机制来同步对共享区域的访问。

　　● 一旦共享内存区域被映射到进程虚拟地址空间，它看起来就像该进程内存空间的其他部分。

　　● 系统通过相似的方式将共享内存区域放置到进程虚拟地址空间 (见48.5)。Linux特有的文件/proc/PID/maps会列出所有类型的共享内存区域的相关信息。

　　● 当不需要将共享内存区域映射到固定地址时，应该确保该区域中的所有引用都是相对偏移量 (因为该区域在不同进程中可能位于不同的虚拟地址)。

　　● 第50章介绍的虚拟内存操作函数能应用于所有类型的共享内存区域。

　　这3种共享内存技术之间也存在一些显著差异：

　　● 因为共享文件映射的内容会与底层文件同步，所以其中的数据在系统重启后依然会被保留。

　　● System V共享内存使用了不兼容传统UNIX I/O模型的键和标识符方案，并且需要使用独立的系统调用和命令。相比之下，POSIX共享内存使用的是名称和文件描述符，所以可以通过很多UNIX系统调用查看和操作。

　　● System V共享内存段的大小在创建时就已固定。相比之下，对于共享文件映射和POSIX共享内存对象，可以使用ftruncate()调整底层对象的大小，并通过munmap()和mmap()重新映射。

　　● 从历史角度，System V共享内存比共享文件映射和POSIX共享内存更普及 (尽管现在大多数UNIX实现都提供了这3种技术)。

54.1 shm_open()

　　函数shm_open()会创建或打开共享内存对象。该函数的参数与open()的参数类似。

　　参数name标识共享内存对象。参数oflag是用于控制shm_open()行为的位掩码，可以包含以下标志：

oflag标志描述
O_CREAT共享内存对象不存在时创建
O_EXCL与O_CREAT一起使用时显式创建共享内存对象
O_RDONLY以只读模式打开
O_RDWR以读写模式打开
O_TRUNC将共享内存对象截断为0字节

　　oflag的用途之一是决定打开或创建共享内存对象。若该参数不包含标志O_CREAT，则打开共享内存对象。若该参数包含标志O_CREAT，则会在共享内存对象不存在时创建。若该参数包含标志O_CREAT和O_EXCL，则会在共享内存对象存在时出错 (EEXIST)。

　　oflag还可用于指定调用进程对共享内存对象进行的访问类型 (指定标志O_RDONLY或O_RDWR)。

　　标志TRUNC会导致成功打开共享内存对象时将其截断为0字节。对于Linux，即使以只读模式打开，这种截断也会生效。但是，SUSv3指出，截断以只读模式打开的共享内存对象将导致未定义的行为。

　　当通过shm_open()创建共享内存对象时，其所有者和所属组取自调用进程的有效ID，并且会根据参数mode设置其权限。可以指定给mode的标志与文件权限标志 (见15.4) 相同。与open()相同，该参数指定的权限受限于进程umask，并且调用shm_open()时总是需要指定该参数。若无需创建共享内存对象，则可以将该参数指定为0。

　　shm_open()返回的文件描述符的close-on-exec标志会被设置 (当进程执行exec()时，这种文件描述符会自动被关闭)。

　　当创建共享内存对象时，其大小为0字节，所以创建完成后通常会在调用mmap()之前通过ftruncate()设置其大小。调用mmap()后可能需要再次通过ftruncate()将共享内存对象调整为预期大小 (见49.3.3)。

　　当扩展共享内存对象时，新增的字节会自动被初始化为0。

　　可以通过对shm_open()返回的文件描述符调用fstat()来获取包含共享内存对象相关信息的结构体stat，这些信息包括大小 (st_size)、权限 (st_mode)、所有者 (st_uid) 和所属组 (st_gid)。SUSv3要求fstat()通过结构体stat返回这些信息 (除了返回这些信息之外，Linux还会通过其他成员返回更多信息)。

　　共享内存对象的权限和所有权分别可以通过fchmod()和fchown()进行修改。

54.2 使用共享内存对象

　　这2个示例展示了如何通过共享内存对象在进程之间传输数据。

　　pshm_wirte.c会将第2个命令行参数作为字符串复制到第1个命令行参数指定的现有共享内存对象。在映射共享内存对象和复制之前，该程序会通过ftruncate()将共享内存对象的长度调整为待复制字符串的长度。

　　pshm_read.c会将第1个命令行参数指定的现有共享内存对象中的字符串打印到标准输出。当调用shm_open()后，该程序会通过fstat()获取共享内存对象的大小，然后在通过mmap()映射该共享内存对象时指定该大小，最后通过write()打印字符串。

54.3 shm_unlink()

　　函数shm_unlink()会取消链接名为name的共享内存对象。这不会影响该共享内存对象的现有映射，但会阻止后续通过shm_open()打开它。一旦所有进程都取消映射该共享内存对象，则它会被删除，并且其内容会丢失。
