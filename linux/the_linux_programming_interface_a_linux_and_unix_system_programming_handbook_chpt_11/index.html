《Linux/UNIX系统编程手册》第11章 系统限制和选项

发表于 2024-12-14 更新于 2025-01-15 分类于 Linux 本文字数： 7.5k

　　每个UNIX实现都对各种系统特性和资源施加了限制，并且提供了/未提供各种标准中定义的选项。但是，限制和选项可能因UNIX实现、运行时环境或文件系统的不同而不同。

　　系统限制和选择可能会影响程序的行为，所以可移植应用程序需要方法来确定限制值和支持的选项。C标准和SUSv3提供了2种用于获取这些信息的方法：

　　● 有些限制和选项可以在编译时确定。例如，int的最大值由硬件架构和编译器设计选择决定。此类限制可能记录在头文件中。

　　● 其他限制和选项在运行时可能有所不同。SUSv3定义了函数sysconf()、pathconf()和fpathconf()来确定这些限制和选项的值。

　　对于SUSv3详细说明的每个限制，SUSv3都要求所有实现支持该限制的上限的最小值，该最小值通常是定义在头文件limits.h中的带有前缀_POSIX_的常量，并且 (通常) 带有后缀_MAX，所以其名称格式是_POSIX_XXX_MAX。 有时，会使用限制的下限的最大值，并且其名称带有后缀_MIN。

　　每个限制都有1个与上限的最小值名称对应的名称，其名称格式为XXX_MAX。实现可以在limits.h中定义名称为XXX_MAX的常量来指定的限制的上限，若定义了该常量，则其值必须不小于上限的最小值。

　　除了详细说明系统资源的限制之外，SUSv3还详细说明了UNIX实现可能支持的各种选项，例如，实时信号、POSIX共享内存、作业控制和POSIX线程。SUSv3仅要求实现支持其中的少部分选项。此外，SUSv3允许实现提供在编译时和运行时检测是否支持特定选项的方法。

　　实现可以通过在头文件unistd.h中定义常量来表示它是否支持对应的SUSv3选项。这类常量都带有表示对应标准的前缀 (例如，_POSIX_和_XOPEN_)，其值如下：

　　● 定义为-1表示不支持对应的选项。实现不能定义该选项相关的头文件、数据类型和函数接口。应用程序可能需要使用#if语句来处理这种可能性。

　　● 定义为0表示可能支持对应的选项。应用程序必须在运行时确定实现是否支持该选项。

　　● 定义为正值表示支持对应的选项。实现会定义该选项相关的头文件、数据类型和函数接口，并且其行为如同SUSv3所说。在很多情况下，SUSv3要求该值为200112L，对应SUSv3获批的年份和月份。与之类似，SUSv4要求该值为200809L。

11.1 系统限制

　　SUSv3将限制分为运行时不变值、路径名可变值和运行时可增加值。

　　● 若limits.h中定义了运行时不变值，则其值是固定的。尽管如此，该值可能是不确定的 (因为取决于可用的内存空间等原因)，此时，limits.h中的定义无效。对于这种情况，应用程序可以通过sysconf()来获取其值。例如，运行时不变值MQ_PRIO_MAX定义了POSIX消息队列中的消息的优先级。SUSv3定义了常量_POSIX_MQ_PRIO_MAX来作为该限制的最小值，并将其指定为32。可以通过调用sysconf(_SC_MQ_PRIO_MAX)来在运行时获取该限制的上限值。

　　● 路径名可变值是是与路径名 (例如，文件、目录和终端) 相关的限制。此类限制可能是常量，也可能因文件系统而异。对于因路径名而异的路径名可变值，应用程序可以使用pathconf()和fpathconf()来获取其值。例如，路径名可变值NAME_MAX定义了特定文件系统中的文件名的最大大小。SUSv3定义了常量_POSIX_NAME_MAX来作为该限制的最小值，并将其指定为14。可以调用pathconf(directory_path, _PC_NAME_MAX)来获取该限制的上限值，其中directory_path是文件系统中的目录。

　　● 对于特定实现，运行时可增加值是上限的固定最小值，并且所有运行该实现的系统提供的此类限制的上限必须不小于该最小值。但是，特定系统可以在运行时增加此类限制的上限值。例如，运行时可增加值NGROUPS_MAX定义了进程的辅助组ID的最大数量。SUSv3定义了常量_POSIX_NGROUPS_MAX来作为该限制的最小值。可以调用sysconf(_SC_NGROUPS_MAX)来在运行时获取该限制的上限值。

　　● 函数getdtablesize()是获取限制OPEN_MAX的值的已过时的替代方法。SUSv2详细说明了该函数 (标记为LEGACY)，但SUSv3没有说明该函数。

　　● 函数getpagesize()是获取限制_SC_PAGESIZE的值的已过时的替代方法。SUSv2详细说明了该函数 (标记为LEGACY)，但SUSv3没有说明该函数。

　　● 头文件stdio.h中定义的常量FOPEN_MAX等同于STREAM_MAX。

　　● NAME_MAX不包含结尾的字符\0，但PATH_MAX包含结尾的字符\0。这种不一致修复了POSIX.1中的早期的不一致 (无法确定PATH_MAX是否包含结尾的字符\0)。将PATH_MAX定义为包含结尾的字符\0表示应用程序会为符合规范的路径名分配PATH_MAX字节。

11.2 在运行时检索系统限制

　　函数sysconf()会返回参数name (unistd.h中定义的带有前缀_SC_的常量) 指定的限制的值。若限制不确定或出错 (只会出现错误EINVAL，表示name无效)，则返回-1。为了区分这2种情况，需要在调用前先将error设置为0，然后在调用完后判断errno。

　　sysconf()的返回值是长整型。SUSv3规定，sysconf()的返回值可以是字符串，但由于实现和使用的复杂性，没有实现这样做。

　　SUSv3要求sysconf()的目标限制在该函数的调用进程的生命周期内是常量。但是，Linux中有几个合理的例外。进程可以在sysconf()返回限制值后调用setrlimit()来修改限制RLIMIT_NOFILE (决定进程能够同时打开的文件描述符的最大数量，对应_SC_OPEN_MAX)、RLIMIT_NPROC (SUSv3没有说明的限制，决定每个用户通过该进程可以创建的进程数，对应_SC_CHILD_MAX) 和RLIMIT_STACK (仅限于Linux 2.6.23及以上版本，决定进程的命令行参数和环境列表的最大字节数，对应_SC_ARG_MAX)。

11.3 在运行时检索文件相关的限制

　　函数pathconf()和fpathconf()都会返回参数name (unistd.h中定义的带有前缀_PC_的常量) 指定的限制的值。两者唯一的区别是指定文件或目录的方式，pathconf()指定的是路径名，fpathconf()指定的是文件描述符。

　　除了11.1介绍的限制之外，以下限制可用于这2个函数：

常量名描述
_PC_NAME_MAX对于目录，这将为该目录下的文件生成1个值 (未指定其他文件类型的行为)
_PC_PATH_MAX对于目录，这将生成此目录的相对路径名的最大长度 (未指定其他文件类型的行为)
_PC_PIPE_BUF对于有名/无名管道，这将生成1个适用于引用的文件的值；对于目录，该值适用于该目录下创建的有名管道 (未指定其他文件类型的行为)

　　当这2个函数返回-1时，处理方法与sysconf()的类似。

　　SUSv3不要求pathconf()和fpathconf()的目标限制在这2个函数的调用进程的生命周期内是常量 (因为在调用进程的运行过程中，文件系统可能以不同的特征卸载和重新挂载)。

11.4 不确定的限制

　　有时，可能会出现系统限制不是由实现限制常量定义的情况，此时，sysconf()或pathconf()会返回-1来表示该限制不确定。对于这种情况，可以使用如下方法来处理：

　　● 当开发可移植应用程序时，可以使用SUSv3指定的上限的最小值 (对应名称为_POSIX_*_MAX的常量)。有时，这种方法可能不可行，因为最小值可能小到不切实际 (例如，_POSIX_PATH_MAX和_POSIX_OPEN_MAX)。

　　● 在某些情况下，实用的解决方法是忽略对限制的检测，并且调用相关系函数。若调用失败，并且errno表示因超出某些系统限制而出错，则可以进行适当的修改，然后再次尝试。例如，大多数UNIX实现对进程的排队的信号的数量施加了限制，一旦达到上限，对该进程继续发送信号会导致错误EAGAIN。对于这种情况，发送进程只需要在等待一段时间，然后再次发送信号。

　　● 编写自己的函数 (或程序) 来预估限制。预估函数在调用sysconf()和pathconf()后，若目标限制不确定，则预估函数返回预估值。尽管这种方法不完美，但在实际中通常是可行的。

　　● 使用autoconf (GNU扩展工具) 等工具来确定各种系统特性和限制是否存在和相关设置。autoconf程序会基于它确定的信息生成可供C程序引用的头文件。

11.5 系统选项

　　当系统选项对应的常量定义为0时，应用程序可以通过函数sysconf()和pathconf() (或fpathconf()) 来在运行时检测实现是否该选项。传递给这些函数的参数name的名称通常与对应的编译时常量的类似，但前缀是_SC_或_PC_。实现必须提供足以执行运行时检测的头文件、常量和函数接口。

　　SUSv3没有说明将未定义的选项对应的常量定义为0或-1时的含义。标准委员会随后决定将这种情况视为将常量定义为-1 (即不支持该选项)，SUSv4显式地说明了这一点。

　　上表列出了部分SUSv3详细说明的选项。第1列是选项对应的编译时常量名 (定义在unistd.h中)，第2列是对应的指定给sysconf() (_SC_*) 或pathconf() (_PC_*) 的参数name的名称。

　　● 对于SUSv3要求提供的选项 (表中用 + 标记)，对应的编译时常量必须为正值。从历史角度，这些选项过去是可选的，但如今是必选的。对于SUSv4，SUSv3中的很多可选的选项成为了必选。尽管SUSv3要求提供这些选项，但有些UNIX实现可能以不符合规范的配置安装。因此，对于可移植应用程序，检测实现是否支持这些选项可能是值得的。

　　● 对于特定选项 (表中用 * 标记)，对应的编译时常量必须非负。
