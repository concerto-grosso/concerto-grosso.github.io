《Linux/UNIX系统编程手册》第47章 System V信号量

发表于 2026-04-03 更新于 2026-04-05 分类于 Linux 本文字数： 10k

　　信号量是内核维护的整数，其值被限制为不小于0。可以对信号量执行各种操作 (即系统调用)，包括将信号量设置为不小于0的值、加减信号量以及等待信号量为0。

　　减少信号量和等待信号量为0会导致调用进程被阻塞。当减少信号量时，内核会阻止任何将信号量减少至小于0的尝试。同样，等待信号量为0的进程在信号量不为0时会被阻塞，直到其他进程将信号量调整为允许操作继续执行的值。下图展示了如何使用信号量来同步2个进程：

　　当使用信号量控制进程的行为时，信号量本身没有意义，其意义由使用它的进程赋予它的关联决定。通常，进程会约定某个规则，将信号量与共享资源 (例如，共享内存区域) 关联起来。信号量还可用于fork()之后同步父子进程等其他用途。

　　所有信号量都有1个关联的结构体semid_ds。

　　各种信号量系统调用都会隐式地更新结构体semid_ds的成员。系统调用semctl()的操作IPC_SET可用于显式地修改该结构体的部分成员。

　　● 成员sem_perm会在创建信号量集时被初始化，其子成员uid、gid和mode可以通过操作IPC_SET更新。

　　● 成员sem_otime会在创建信号量集时被设置为0。当后续每次成功调用semop()或由于操作SEM_UNDO而更新信号量时，该成员会被更新为当前时间 (自纪元以来的秒数)。

　　● 成员sem_ctime会在创建信号量集以及后续成功执行操作IPC_SET、SETALL或SETVAL时被设置为当前时间 (对于某些UNIX实现，操作SETALL或SETVAL不会修改该成员)。

　　● 成员sem_nsems会在创建信号量集时被初始化为信号量集中的信号量数。

　　SUSv3要求结构体semid_ds包含以上所有成员。某些UNIX实现还包含额外的非标准成员。对于Linux 2.4及以上版本，成员sem_nsems的类型是unsigned long。SUSv3将该成员的类型指定为unsigned short，并且Linux 2.2和大多数UNIX实现也是这样定义。

　　当进程因信号量的的状态无法满足执行该进程所请求的信号量操作而一直被阻塞时，就会导致该进程饥饿。

47.1 System V信号量概览

　　使用System V信号量的一般步骤通常如下：

　　● 使用semget()创建或打开信号量集。

　　● 使用semctl()的操作SETVAL或SETALL初始化信号量集中的信号量 (该步骤应该仅由1个进程完成)。

　　● 使用semop()操作信号量。使用信号量的进程通常会通过这些操作来表示共享资源的获取和释放。

　　● 当所有进程都不再需要信号量集时，应该使用semctl()的操作IPC_RMID将其删除 (该步骤应该仅由1个进程完成)。

　　大多数操作系统都提供了某种类型的信号量原语供应用程序使用。但是，System V信号量是以称为信号量集的组进行分配，进而导致情况变得相当复杂。使用semget()创建信号量集时需要指定其中的信号量数。虽然1次操作单个信号量很常见，但semop()允许以原子方式对同一信号量集中的多个信号量执行多个操作。

　　这个示例展示了各种System V信号量系统调用的用法。 该程序可以以2种模式运行：

　　● 若提供了2个命令行参数，该程序会其解释为现有信号量集的标识符以及对将该信号量集中的第1个信号量执行的操作。为了监控执行的操作，该程序会在执行操作前后打印相关信息。这些信息都以进程ID开头，以便区分该程序的多个实例的输出。

47.2 创建和打开信号量集

　　系统调用semget()会创建信号量集或获取现有信号量集的标识符。

　　参数key是键 (见45.2)。当使用semget()创建信号量集时，参数nsems用于指定新号量集中的信号量数；当使用semget()获取现有信号量集的标识符时，参数nsems必须不大于该信号量集中的信号量数 (否则，导致错误EINVAL)。信号量集中的信号量数只能在创建时指定，后续无法修改。参数semflg是位掩码，用于指定新信号量集的权限或检查现有信号量集的权限，并且可以包含标志IPC_CREAT和/或IPC_EXCL。

　　若成功调用，则semget()会返回新信号量集或现有信号量集的标识符。后续以单个信号量为目标的系统调用必须同时指定信号量集标识符和信号量在信号量集中的编号。信号量集中的信号量编号从0开始。

47.3 信号量控制操作

　　系统调用semctl()会对信号量集或其中的某个信号量执行各种控制操作。

　　参数semid是信号量集标识符。对于仅对1个信号量执行的操作，参数semnum用于标识信号量集中的特定信号量。对于其他操作，参数semnum会被忽略，所以可以将其指定为0。参数cmd用于指定操作。特定操作会使用第4个参数，该参数是共用体，并且需要在程序中显式地定义该共用体 (这里将其定义在头文件semun.h中供示例程序使用)。

　　虽然在标准头文件中定义共用体semun是合理的，但SUSv3要求程序员显式地定义它。尽管如此，有些UNIX实现在头文件sys/sem.h中提供了该共用体的定义。较旧的glibc版本 (2.0及以下版本) 也提供了该共用体的定义。为了符合SUSv3，较新的glibc版本不再提供该定义，并在头文件sys/sem.h中将宏_SEM_SEMUN_UNDEFINED定义为1来指示这一事实 (即glibc编译的程序可以通过该宏来确定是否需要显式地定义该共用体)。

47.3.1 通用控制操作

　　以下信号量控制操作同样适用于其他类型的System V IPC对象 (对于这些操作，参数semnum会被忽略)：

　　● IPC_RMID会立即删除信号量集及其关联的结构体semid_ds。所有因等待该信号量集而被阻塞在semop()调用的进程会立即被唤醒并出错 (EIDRM)。此外，参数arg会被忽略。

　　● IPC_STAT会将信号量关联的结构体semid_ds的副本存储在arg.buf指向的缓冲区中。

　　● IPC_SET会使用arg.buf指向的缓冲区中的值更新信号量集关联的结构体semid_ds。

47.3.2 检索和初始化信号量值

　　以下操作会检索 (需要信号量的读取权限) 或初始化 (需要信号量的写入权限) 信号量集中的指定或全部信号量的值：

　　● 操作GETVAL会让semctl()返回semid指定的信号量集中的第semnum个信号量的值。此外，参数arg会被忽略。

　　● 操作SETVAL会让semctl()将semid指定的信号量集中的第semnum个信号量初始化为arg.val。

　　● 操作GETALL会让semctl()将semid指定的信号量集中的所有信号量的值存储在数组arg.array。程序员必须确保该数组足够大 (信号量集中的信号量数可以通过操作IPC_STAT获取)。

　　● 操作SETALL会让semctl()使用数组arg.array初始化semid指定的信号量集中的所有信号量。此外，参数semnum会被忽略。

　　若进程正在等待对由操作SETVAL或SETALL修改的信号量执行操作，并且所做的修改允许该操作继续执行，则内核会唤醒该进程。

　　使用操作SETVAL或SETALL修改信号量会清空所有进程中的该信号量的撤销条目。

　　当进程使用操作GETVAL或GETALL返回的信息时，该信息可能已过时。所有依赖于这些操作返回的信息保持不变的程序都可能受到TOCTTOU竞争条件的影响 (见38.4)。

47.3.3 检索信号量级信息

　　以下操作会返回semid指定的信号量集中的第semnum个信号量的相关信息 (这些操作都需要信号量集的读取权限，并且会忽略参数arg)：

　　● 指定了操作GETPID后，semctl()会最后对信号量调用semop()的进程的进程ID，该值称为sempid值。若还没有进程对信号量调用调用semop()，则返回0。

　　● 指定了操作GETPID后，GETNCNT()会当前正在等待信号量的值增加的进程的数量，该值称为semncnt值。

　　● 指定了操作GETZCNT后，GETNCNT()会当前正在等待信号量的值为0的进程的数量，该值称为semzcnt值。

　　与操作GETVAL和GETALL相同，这3个操作返回的信息可能在进程使用它时已经过时。

47.5 信号量操作

　　系统调用semop()会对参数semid标识的信号量集中的信号量执行操作。

　　参数sops是由结构体sembuf组成的数组，该数组指定待执行的操作。参数nsops指定该数组的大小 (该数组包含至少1个元素)。

　　对于结构体sembuf，成员sem_num标识信号量，成员sem_op指定操作，成员sem_flg是操作标志。

　　● 若sem_op大于0，则会将信号量加上该值。因此，其他等待信号量增加的进程可能会被唤醒并执行操作。调用进程必须拥有目标信号量的修改 (写入) 权限。

　　● 若sem_op等于0，则会判断信号量是否为0。若是，则操作会立即完成；否则，调用进程会被阻塞，直到信号量为0。调用进程必须拥有目标信号量的读取权限。

　　● 若sem_op小于0，则会将信号量减去该值的绝对值。若信号量的当前值不小于该值的绝对值，则操作会立即完成；否则，调用进程会被阻塞，直到信号量的值增加至允许操作继续执行的值 (不会导致信号量小于0)。调用进程必须拥有目标信号量的修改权限。

　　从语义角度，增加信号量相当于释放资源，减少信号量相当于为当前进程 (独占地) 保留资源。当减少信号量时，若信号量的当前值太小 (其他进程已占用资源)，则进程会被阻塞。

　　当进程因调用semop()而被阻塞时，它会一直被阻塞，直到出现以下情况之一：

　　● 其他进程将信号量修改为允许操作继续执行的值。

　　● 信号处理函数中断semop()调用。对于这种情况，semop()会出现错误EINTR (如前所述，semop()在被信号处理函数之后永远不会自动重启)。

　　● 其他进程删除了semid指定的信号量。对于这种情况，semop()会出现错误EIDRM。

　　可以在信号量对应的结构体sem_buf的成员sem_flg中指定标志IPC_NOWAIT来防止semop()阻塞进程。对于这种情况，semop()会出错 (EAGAIN)。

　　通常，每次调用semop()只会操作1个信号量。实际上，可以通过semop()操作多个信号量。关键点在于多个操作是以原子方式执行，所以semop()要么执行所有操作，要么阻塞调用进程来等待所有操作可以同时执行。

　　所有系统都是按照数组sops的顺序执行操作，并且部分应用程序依赖于这种行为，但只有少数系统记录了这种行为。SUSv4添加了明确要求这种行为的说明。

　　这个示例展示了如何使用semop()对信号量集中的3个信号量执行操作。对信号量0和2执行的操作可能无法立即执行 (取决于信号量的值)。若对信号量0执行的操作无法立即执行，则其他操作也不会执行，并且调用进程会被阻塞。另一方面，若对信号量0执行的操作能够立即执行，但对信号量2执行的操作无法立即执行，则这些操作会因指定了标志IPC_NOWAIT而无法执行，semop()会立即出错 (EAGAIN)。

　　系统调用semtimedop()与semop()类似，但可以通过参数timeout指定阻塞时间的上限。若操作未在指定的时间之前完成，则该系统调用会出错 (EAGAIN)。若timeout为NULL，则该系统调用与semop()完全相同。

　　semtimedop()提供了比setitimer()与semop()的组合更高效的信号量操作超时时间设置方法。该系统调用能为频繁执行这种操作的应用程序带来显著的性能提升。但是，SUSv3没有说明该系统调用，并且只有部分UNIX实现提供了该系统调用。

　　semtimedop()作为Linux 2.6的新特性出现，随后向后移植到Linux 2.4.22及以上的2.4版本。

　　● semnum+value表示将信号量semnum增加value。

　　● semnum-value表示将信号量semnum减少value。

　　● semnum＝0表示判断信号量semnum是否为0。

　　所有操作的结尾都可以加上字母n和/或u。n表示为该操作指定标志IPC_NOWAIT。u表示为该操作指定标志SEM_UNDO。

47.6 信号量撤销值

　　若进程在修改信号量后终止，则信号量默认会维持修改后的值。但是，这可能会为其他使用该信号量的进程带来问题——等待已终止的进程撤销其修改。

　　为了避免这种问题，可以在通过semop()修改信号量时指定标志SEM_UNDO。当指定了该标志后，内核会记录信号量操作的效果，并在进程终止时撤销该操作。

　　内核不会记录所有指定了标志SEM_UNDO的操作。相反，内核会将每个进程对每个信号量的修改的总和记录在对应的semadj值。当进程终止时，内核只需要根据semadj值修改信号量。

　　当使用semctl()的操作SETVAL或SETALL设置信号量时，使用该信号量的所有进程中的对应的semadj (信号量调整) 值会被清空 (设置为0)。这是合理的，因为以绝对方式设置信号量会破坏与semadj值中维护的历史记录相关的值。

　　使用fork()创建的子进程不会进程继承父进程的semadj值 (子进程撤销父进程的信号量操作没有意义)。另一方面，semadj在exec()期间会被保留。

　　从Linux 2.6开始，若使用clone()创建的进程 (线程) 时指定了标志CLONE_SYSVSEM，则会共享semadj值。这种共享是符合POSIX线程实现所必需的。NPTL实现就使用了该标志来实现函数pthread_create()。

　　以下shell会话展示了对2个信号量执行操作的影响 (只有1个操作指定了标志SEM_UNDO)：

　　首先，创建包含2个信号量的信号量集 (这里省略了信号量初始化)。然后，将这2个信号量加1，对信号量0的操作指定了标志SEM_UNDO。最后，打印信号量集相关信息 (对信号量0的操作被撤销，对信号量1的操作被保留)。

　　综上所述，标志SEM_UNDO并没有表面上看起来那样有用，原因有2个。首先，修改信号量通常对应获取或释放共享资源，仅使用该标志不足以让多进程应用程序在进程意外终止时恢复。除非进程在终止时自动会将共享资源状态恢复到一致状态 (这在很多情况下不太可能)，否则仅撤销信号量操作不足以让应用程序恢复正常。其次，进程终止时撤销操作可能无法执行。假设信号量的初始值为0，考虑以下场景：

　　1) 进程A将信号量加2，并且该操作指定了标志SEM_UNDO。

　　2) 进程B将信号量减1，所以信号量当前为1。

　　3) 进程A终止。

　　此时，无法撤销第1步的进程A的信号量操作，因为信号量的值太小。3种可能的解决方法如下：

　　● 强制阻塞进程A，直到能够撤销其信号量操作。

　　● 尽管可能地减少信号量 (即减少至0)，然后进程A终止。

　　● 进程A直接终止，而不撤销其信号量操作。

　　第1种方法不可行，因为这可能导致即将终止的进程A永远被阻塞。Linux采用的是第2种方法，有些UNIX实现采用的是第3种方法。SUSv3没有说明如何处理这种情况。

　　试图将信号量增加至大于信号量最大值的撤销操作会导致异常情况。在这种情况下，内核总是执行撤销 (将信号量的值增加至大于信号量最大值的值)。

47.7 实现二进制信号量协议

　　因为信号量可以被调整为 (限制之内的) 任意值，并且信号量的分配和释放基于信号量集，所以System V信号量API比较复杂。这2个特性提供的功能超过了常规应用程序的需求，所以实现一些基于System V信号量的简单协议 (API) 会很有用。

　　二进制信号量是常用的协议之一。二进制信号量有2个值：可用 (空闲) 和已占用 (使用中)。二进制信号量定义了2个操作：预留和释放。

　　● 预留操作表示试图预留信号量以供独占使用。若信号量已被其他进程占用，则进程会被阻塞，直到信号量被释放。

　　● 释放操作表示释放已占用的信号量供其他进程使用。

　　在计算机科学学术领域，这2个操作通常称为P和V，这是它们在荷兰语中的首字母。这些术语由已故的荷兰计算机科学家Edsger Dijkstra提出，他完成了关于信号量的很多早期研究。术语down (减少信号量) 和up (增加信号量) 也能用于表示这2个操作。POSIX将这2个操作称为wait和post。

　　有时，还会定义第3个操作：条件预留。该操作表示试图以非阻塞方式预留信号量以供独占使用。若信号量已被占用，则该操作会立即返回表示信号量不可用的状态。

　　头文件binary_sems.h的外部变量bsUseSemUndo和bsRetryOnEintr分别用于控制调用semop()时是否指定标志SEM_UNDO以及semop()被信号处理函数中断时是否重启。

　　这个示例展示了二进制信号量函数的实现。这些函数都只有2个参数，分别用于指定信号量集和该信号量集中信号量的编号 (这些函数不会创建和删除信号量集，也不会处理TOCTTOU竞争条件)。

47.8 信号量限制

　　大多数UNIX实现都对System V信号量施加了各种限制。

　　Linux的信号量限制如下 (受限制影响的系统调用以及达到限制时产生的错误信息在括号中注明)：

　　● 限制SEMAEM是semadj值中可以记录的最大值 (semop()和ERANGE)。

　　● 限制SEMMNI是可以创建的信号量集标识符的数量的系统级限制 (semget()和ENOSPC)。

　　● 限制SEMMSL是信号量集中的信号量数的最大值 (semget()和EINVAL)。

　　● 限制SEMMNS是所有信号量集中的信号量的总数的系统级限制，该限制的默认值是限制SEMMNI和SEMMSL的默认值的乘积 (semget()和ENOSPC)。

　　● 限制SEMOPM是调用semop()时可以指定的操作数的最大值 (semop()和E2BIG)。

　　● 限制SEMVMX是信号量值的最大值 (semop()和ERANGE)。

　　这些限制也出现在大多数UNIX实现中。某些UNIX实现还施加了以下限制：

　　● 限制SEMMNU是信号量撤销结构体 (用于存储semadj值) 总数的系统级限制。

　　● 限制SEMUME是信号量撤销结构体中的撤销条目数的最大值。

　　当系统启动时，信号量限制会被设置为默认值。这些默认值可能因内核版本而异 (有些内核发行版使用不同于原版的默认值)。限制SEMVMX和SAMAEM固定为32767，并且无法被修改。部分其他信号量限制可以通过修改特定于Linux的文件/proc/sys/kernel/sem中的值来修改。该文件包含4个由空格分隔的数字，从左到右分别定义了限制SEMMSL、SEMMNS、SEMOPM和SEMMNI。对于基于x86-32架构的Linux 2.6.31，这些限制的默认值如下：

47.9 System V信号量的缺点

　　System V信号量存在很多与消息队列相同的缺点 (见46.9)，包括以下缺点：

　　● 信号量由标识符引用，而非其他大多数UNIX I/O机制所使用的文件描述符。

　　● 键的使用 (而非文件名) 会增加编程复杂度。

　　● 创建和初始化信号量使用不同的系统调用，所以有时可能需要额外的操作来避免初始化信号量时出现的竞争条件。

　　● 内核不会为引用信号量集的进程计数，所以难以确定何时可以安全地删除信号量集以及确保未使用的信号量集已被删除。

　　● 信号量API过于复杂。程序通常只操作1个信号量，同时操作信号量集中的多个信号量的功能略显多余。

　　● 信号量操作存在各种限制。这些限制是可配置的，但当应用程序需要在默认限制范围之外运行时，安装应用程序就需要额外的操作。

　　但是，不同于System V消息队列，System V信号量的替代方案更少，所以其使用场景也更多。记录锁可用于替代其部分用途。此外，从内核2.6开始，Linux支持使用POSIX信号量来同步进程。
