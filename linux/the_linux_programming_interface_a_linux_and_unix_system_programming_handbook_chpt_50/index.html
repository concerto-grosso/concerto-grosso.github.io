50.1 mprotect()

　　系统调用mprotect()会将从地址addr开始length字节内的虚拟内存页的保护修改为参数prot指定的保护。

　　addr必须是系统页大小的整数倍 (SUSv3要求该参数必须按页对齐。但是，SUSv4指出，实现可以要求该参数按页对齐)。由于保护是设置在整页上的，所以length会向上取整到最近接的系统页大小的整数倍。

　　prot是位掩码。该参数要么是PROT_NONE，要么至少包含PROT_READ、PROT_WRITE和PROT_EXEC中的1个标志 (或运算|)。这些标志对mprotect()的意义与对mmap()的意义相同 (见49.1)。

50.2 内存锁

　　对于某些应用程序，锁定进部分或全部的进程虚拟内存很有用，因为这样可以保证它始终位于物理内存中。除了性能提升之外，另一个锁定内存的原因是安全。若包含敏感信息的虚拟内存页永远不会被换出，则其副本永远不会被写入到磁盘。当虚拟内存页被写入到磁盘后，从理论角度，后续可以直接从磁盘设备读取该页。攻击者可以通过运行占用大量内存的程序来强制其他进程的内存被换出到磁盘，从而制造类似的情况。从交换空间中读取信息甚至可以在进程终止后进行，因为内核不会保证交换空间中的数据被清零 (读取交换空间通常需要特权)。

　　笔记本电脑和某些桌面系统的睡眠模式会将系统RAM的副本存储在磁盘，无论是否锁定内存。

50.2.1 资源限制RLIMIT_MEMLOCK

　　资源限制RLIMIT_MEMLOCK的软限制是进程可以锁定在内存的字节数上限。在Linux 2.6.9之前，只有特权进程 (能力CAP_IPC_LOCK) 才能锁定内存 (受限于该资源限制)。从Linux 2.6.9开始，由于内存锁模型的修改，特权进程可以锁定的内存字节数不再受到限制，而非特权进程可以锁定少量内存 (受限于该资源限制)。

　　该资源限制的软限制和硬限制默认都是系统页大小的8倍。该资源限制会影响mlock()、mlockall()、带有标志MAP_LOCKED的mmap()以及shmctl()的操作SHM_LOCK (虚拟内存是以页为单位进行管理的，内存锁会应用于整个页，所以执行资源限制检查时该资源限制会向下取整到最接近的系统页大小的整数倍)。

　　尽管资源限制RLIMIT_MEMLOCK只有1个值，但它定义了2个不同的限制：

　　● 对于mlock()、mlockall()和带有标志MAP_LOCKED的mmap()，该资源限制定义的是进程可以锁定的虚拟地址空间字节数的进程级限制。

　　● 对于shmctl()的操作SHM_LOCK，该资源限制定义的是所有实际用户ID相同的进程可以锁定的共享内存段字节数的用户级限制。

　　该资源限制对System V共享内存段具有不同语义的原因是分离后的共享内存段仍然可以继续存在 (只有显式通过shmctl()的操作IPC_RMID删除共享内存段，并且所有进程都将其分离后，它才会被删除)。

50.2.2 mlock()和munlock()

　　系统调用mlock()会锁定调用进程虚拟地址空间中从地址addr开始length字节内的页。不同于其他内存相关系统调用，addr无需按页对齐。但是，SUSv3指出，实现可以要求addr是系统页大小的整数倍，可移植应用程序应该确保在调用mlock()和munlock()时将该参数指定为系统页大小的整数倍。

　　内存锁应用于整页，所以锁定区域的起始地址是addr向下取整到最接近的系统页大小的整数倍后的值，结束地址是大于addr + length的下一个页边界 (例如，对于页大小是4096字节的系统，mlock(2000, 4000)会锁定字节0到8191)。

　　成功调用mlock()会确保指定范围内的所有页被锁定并驻留在物理内存中。当物理内存不足以锁定所有请求页或请求超过了资源限制RLIMIT_MEMLOCK，则mlock()会出错。

　　系统调用munlock()会执行与mlock()相反的任务，并且参数的用法与mlock()的参数相同。解锁页不会保证页不再驻留在内存中，页仅在响应其他进程的内存需求时才会从RAM中被删除。除了调用munlock()之外，当进程终止、通过munmap()取消映射已锁定的页或通过带有标志MAP_FIXED的mmap()调用覆盖已锁定的页时，也会解锁页。

　　可以通过Linux特有的文件/proc/PID/status中的字段VmLck来获取进程当前锁定了多少内存。

　　fork()创建的子进程不会继承内存锁，并且exec()期间内存锁也不会得到保留。

　　当多个进程共享一组页时，只要至少1个进程锁定了这些页，它们就会保持锁定状态。对于单个进程，内存锁不会嵌套。当进程多次对同一虚拟地址范围调用mlock()时，只会设置1个锁，只需要调用1次munlock()就可以删除该锁。另一方面，若进程使用mmap()将同一组页 (即同一文件) 映射到虚拟地址空间的多个位置，并锁定这些映射，则这些页将一直保持锁定状态，直到所有映射都被解除映射。

　　shmctl()的操作SHM_LOCK的语义不同于mlock()和munlockall()的语义。

　　● 执行操作SHM_LOCK后页只会在后续引用而导致的缺页换入时才会被锁定，而mlock()和mlockall()会在返回之前将指定页缺页换入并锁定。

　　● 操作SHM_LOCK设置的是共享内存段的属性，而非进程属性 (所以文件/proc/PID/status中的字段VmLck不包含进程附加的通过该操作锁定的System V共享内存段)。这表示，一旦页被缺页换入，它就会一直驻留在内存，即使不存在附加它的进程。相比之下，对于通过mlock()和mlockall()锁定在内存的区域，只要至少有1个进程持有该区域的锁，它就会保持锁定状态。

50.2.3 mlockall()和munlockall()

　　系统调用mlockall()会根据位掩码参数flags锁定调用进程虚拟地址空间中当前已映射的所有页和/或所有未来将映射的页，该参数需要包含标志MCL_CURRENT和MCL_FUTURE中的至少1个标志。

　　● 标志MCL_CURRENT表示锁定调用进程虚拟地址空间中当前已映射的所有页，包括当前为程序的文本段和数据段、内存映射以及栈分配的页。

　　● 标志MCL_FUTURE表示锁定所有后续将映射到调用进程虚拟地址空间的页 (例如，这些页可能是通过mmap()或shmat()映射的共享内存区域的一部分，或者向上增长的堆或向下增长的栈的一部分)。当指定了该标志后，后续的mmap()、sbrk()或malloc()等内存分配操作可能因系统耗尽可以分配给进程的RAM或达到资源限制而出错或栈增长生成信号SIGSEGV。

　　关于使用mlock()创建的内存锁的约束、生命周期和继承的规则也适用于通过mlockall()创建的内存锁。

　　系统调用munlockall()会解锁调用进程的所有页，并撤销任何先前mlockall(MCL_FUTURE)调用的效果。

　　在Linux 2.6.9之前，只有特权进程 (能力CAP_IPC_LOCK) 才能调用munlockall() (不一致的是，非特权进程能调用munlock())。从Linux 2.6.9开始，非特权进程也能调用该系统调用。

50.3 mincore()

　　系统调用mincore()会返回调用进程虚拟地址空间中从地址addr开始length字节内的页的内存驻留信息，结果存储在参数vec指向的缓冲区。

　　addr必须按页对齐。由于返回的是整页的信息，所以length实际上会向上取整到最近接的系统页大小的整数倍。vec的必须是(length + PAGE_SIZE - 1) / PAGE_SIZE)字节的数组 (对于Linux，vec的类型是unsigned char *。对于有些UNX实现，vec的类型是char *)。若页驻留在内存中，则vec中的对应字节的最低有效位会被设置 (某些UNIX实现还会设置其他位，所以可移植应用程序应该仅判断该位)。

　　mincore()返回的信息可能在该调用之后和检查vec元素之前发生变化。只有通过mlock()和mlockall()锁定的页才会始终驻留在内存中。

　　在Linux 2.6.21之前，各种实现问题导致mincore()不会正确返回私有映射和非线性映射的内存驻留信息。

50.4 madvise()

　　系统调用madvise()用于告知内核调用进程可能使用从地址addr开始length字节内的页，从而提高应用程序性能 (内核可以使用该信息来提高在底层页上执行的文件I/O的效率)。Linux从2.4开始提供该系统调用。

　　addr必须按页对齐，length会向上取整到最接近的系统页大小的整数倍。参数advice是以下值之一 (后续版本的Linux添加了更多非标准advice值)：

　　● MADV_NORMAL表示执行默认行为，即页是成簇传输的 (系统页大小的较小整数倍)。这会导致一些预读和后读现象。

　　● MADV_RANDOM表示该区域的页会被随机访问 (预读不会带来任何好处)。因此，内核在每次读取时应尽可能获取最少的数据。

　　● MADV_SEQUENTIAL表示该区域内的页将被顺序访问1次。因此，内核可以积极地预读，并在访问页后迅将其释放。

　　● MADV_WILLNEED表示提前读取该区域的页，以便将来访问 (其效果类似于Linux特有的系统调用readahead()和posix_fadvise()的操作POSIX_FADV_WILLNEED)。

　　● MADV_DONTNEED表示调用进程不再需要该区域的页驻留在内存中。该值的具体效果因UNIX实现而异。对于Linux，该值表示显式丢弃私有映射区域的页 (对这些页的修改将会丢失)，对应的虚拟内存地址访问仍然可以被访问，但每次访问其中的页会导致页故障，页将会被重新初始化 (取自对应的映射文件或被填充为0)。对于共享映射区域，该值可能会丢弃修改后的页，具体取决于硬件架构 (x86-32不会出现这种行为)。有些UNIX实现的行为与Linux的行为相同。但是，对于部分UNIX实现，该值只会通知内核在必要的情况下可以换出该区域的页。可移植应用程序不应该依赖于该值的Linux语义。

　　大多数UNIX实现都提供了madvise()的某个版本，通常至少允许指定以上advice值。但是，SUSv3将该API标准化为不同的名称posix_madvise()，并在对应的advice值前加上了前缀POSIX_。posix_madvise()的glibc实现 (2.2或更高版本) 基于madvise()，但并非所有UNIX实现都提供了该替代接口。

　　SUSv3指出，posix_madvise()不应该影响程序的语义。但是，在glibc 2.7之前，POSIX_MADV_DONTNEED基于madvise()的MADV_DONTNEED。从glibc 2.7开始，POSIX_MADV_DONTNEED没有任何效果 (不会影响程序的语义)。
