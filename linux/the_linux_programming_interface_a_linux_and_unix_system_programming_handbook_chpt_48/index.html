《Linux/UNIX系统编程手册》第48章 System V共享内存

发表于 2026-04-17 更新于 2026-04-18 分类于 Linux 本文字数： 7.6k

　　System V共享内存允许多个进程共享物理内存的同一区域 (通常称为段)。因为共享内存段是进程用户空间内存的一部分，所以基于共享内存的IPC无需内核介入。相比无名管道和消息队列等技术，共享内存的速度更快。另一方面，没有内核的介入表示需要某种同步进程的方法。

　　在内存映射术语中，内存区域是被映射到某个地址。在System V术语中，共享内存段时被附加到某个地址。两者是等效的，术语之间的差异源于它们的起源不同。

　　使用共享内存段的一般步骤通常如下：

　　● 通过shmget()创建共享内存段或获取现有共享内存段的标识符。

　　● 通过shmat()附加共享内存段 (让共享内存段成为调用进程虚拟内存的一部分)。

　　● 此时，共享内存可以像程序的任何其他内存一样被使用。为了引用共享内存，程序需要使用shmat()返回的地址 (共享内存段在进程虚拟地址空间中的起始地址)。

　　● 通过shmdt()分离共享内存段。当调用该系统调用后，进程将无法再访问共享内存。该步骤是可选的，因为进程终止时会自动分离共享内存段。

　　● 通过shmctl()删除共享内存段。当且仅当所有进程都分离共享内存段后，该内存段才会被删除 (该步骤应该仅由1个进程完成)。

　　所有共享内存段都有1个关联的结构体shmid_ds。

　　各种共享内存系统调用都会隐式地更新结构体shmid_ds的成员。shmctl()的操作IPC_SET可用于显式地修改该结构体的部分成员。

　　● 成员shm_perm会在创建共享内存段时被初始化，其子成员uid、gid和mode的低9位可以通过操作IPC_SET更新。除了常规权限位之外，子成员mode还包含2个只读位掩码标志：SHM_DEST和SHM_LOCKED，分别表示该内存段是否被标记为待删除 (shmctl()的操作IPC_RMID) 和是否被锁定在内存中 (shmctl()的操作SHM_LOCK)。SUSv3没有说明这2个标志，并且只有部分UNIX实现提供等效标志 (名称不同)。

　　● 成员shm_segz会在创建共享内存段时会设置为请求的共享内存段大小 (以字节为单位)。共享内存的分配以页为单位，所以该内存段的实际大小可能大于该值。

　　● 成员shm_atime和shm_dtime会在创建共享内存段时被设置为0。shm_atime会在进程附加该内存段时被设置为当前时间，shm_dtime会在进程分离该内存段时被设置为当前时间。

　　● 成员shm_ctime会在创建共享内存段以及后续成功执行操作IPC_SET时被设置为当前时间。

　　● 成员shm_cpid会在创建共享内存段时被设置为创建进程的进程ID。

　　● 成员shm_lpid会在创建共享内存段时被设置0，并且后续成功调用shmat()或shmdt()时会将该成员设置为调用进程的进程ID。

　　● 成员shm_nattch会记录当前已附加共享内存段的进程数。该成员会在创建共享内存段时被初始化为0，并且随着每次成功调用shmat()或shmdt()而增减。该成员的数据类型shmatt_t是无符号整数类型，SUSv3要求该数据类型的大小至少为unsigned short。Linux将该数据类型定义为unsigned long。

　　SUSv3要求结构体shmid_ds包含以上所有成员。部分UNIX实现还包含额外的非标准成员。

48.1 创建或打开共享内存段

　　系统调用shmget()会创建新共享内存段或获取现有共享内存段的标识符。新共享内存段的内容会被初始化为0。

　　参数key是键 (见45.2)。当使用shmget()创建共享内存段时，参数size用于指定预期字节数。内核以系统页为单位分配共享内存，所以新共享内存段的实际大小是size向上取整到页大小的整数倍后的值。当使用shmget()获取共享内存段标识符时，参数size不会影响共享内存段，但必须不大于该内存段的大小。参数shmflg用于指定新共享内存段的权限或检查现有共享内存段的权限，并且可以包含标志IPC_CREAT和/或IPC_EXCL。

　　Linux允许在shmflg中指定非标准标志SHM_HUGETLB (Linux 2.6或更高版本) 和SHM_NORESERVE (Linux 2.6.15或更高版本)。

　　● 特权进程 (能力CAP_IPC_LOCK) 可以使用标志SHM_HUGETLB来创建使用巨页的共享内存段。巨页是很多现代硬件架构提供的特性之一，用于使用很大的页大小来管理内存 (例如，x86-32允许使用4MB页来替代4KB页)。对于内存很大的系统，当程序需要大块内存时，使用巨页可以减少内存管理单元的转换后备缓冲区中所需的条目数。

　　● 标志SHM_NORESERVE对shmget()的作用与标志MAP_NORESERVE对mmap()的作用相同。

　　当成功调用shmget()时，会返回新共享内存段或现有共享内存段的标识符。

48.2 使用共享内存

　　系统调用shmat()会将参数shmid标识的共享内存段附加到调用进程的虚拟地址空间。参数shmaddr以及在位掩码参数shmflg中是否指定标志SHM_RND决定如何附加共享内存段：

　　● 当shmaddr为NULL时，共享内存段会被附加到由内核选择的合适地址 (这是附加共享内存段的首选方法)。

　　● 当shmaddr不为NULL时，若未指定标志SHM_RND，则共享内存段会被附加到地址shmaddr，该地址必须是系统页大小的整数倍 (否则，导致错误EINAL)；若指定了标志SHM_RND，则共享内存段会被附加到shmaddr向下取整到常量SHMLBA (共享内存低边界地址) 的整数倍的地址。该常量是系统页大小的整数倍。对于有些架构，将共享内存段附加到SHMLBA的整数倍地址是必要的，这样既能提高CPU缓存性能，又能避免同一共享内存段的不同附加点在CPU缓存中产生不一致的视图 (对于x86架构，该常量就是系统页大小，所以不会出现这种不一致的情况)。

　　应该尽量将参数shmaddr指定为NULL，原因如下：

　　● 非NULL的shmaddr会降低应用程序的可移植性 (某个UNIX实现中的有效地址在其他UNIX实现中可能无效)。

　　● 当地址shmaddr已被使用时，试图附加共享内存段会出错 (例如，若应用程序或它调用的库函数已在该地址附加了共享内存段或创建了内存映射，则可能发生这种情况)。

　　当成功调用shmat()时，会返回附加共享内存段的地址。该地址可以像普通C指针一样被使用，该内存段看起来就像进程虚拟内存的其他部分。shmat()返回的地址通常会被赋值给某个程序员定义的结构体指针，以便于将结构体映射到该内存段。

　　为了附加只读共享内存段，需要在参数shmflg中指定标志SHM_RDONLY。试图写入只读共享内存段会导致段故障 (信号SIGSEGV)。若未指定该标志，则共享内存段既是可读写的。

　　为了附加共享内存段，进程需要其读取和写入权限。若指定了标志SHM_RDONLY，则进程仅需要其读取权限。

　　进程可以多次附加同一共享内存段，甚至可以分别以只读和读写方式附加。所有附加点的共享内存段的内容都是相同的，因为进程虚拟内存页表中的不同条目引用的是同一物理内存页。

　　除了以下标志之外，还可以在参数shmflg中指定标志SHM_REMAP。此时，参数shmaddr必须不为NULL。该标志会让shmat()会替换从地址shmaddr开始的长度为共享内存段长度范围内的所有现有共享内存段或内存映射。若未指定该标志，则将共享内存段附加到正在使用的地址范围会导致错误EINVAL。该标志是非标准Linux扩展。

标志描述
SHM_RDONLY附加只读共享内存段
SHM_REMAP替换位于地址shmaddr处的所有现有映射
SHM_RNDshmaddr向下取整到SHMLBA的整数倍

　　系统调用shmdt()会从调用进程的虚拟地址空间中分离位于地址shmaddr处的共享内存段。

　　通过fork()创建的子进程会继承父进程附加的共享内存段。所有附加的共享内存段都会在exec()期间或进程终止时被分离。

48.3 共享内存控制操作

　　系统调用shmctl()会对参数shmid标识的共享内存段执行各种控制操作。参数cmd用于指定操作。只有操作IPC_STAT和IPC_SET才需要使用参数buf，执行其他操作时应该将该参数指定为NULL。

　　以下共享内存段控制操作同样适用于其他类型的System V IPC对象：

　　● IPC_RMID会将共享内存段及其关联的结构体shmid_ds标记为待删除。若当前没有进程附加该内存段，则立即删除；否则，在所有进程都分离该内存段后才删除。对于Linux，若共享内存段被标记为待删除，并且存在未分离该内存段的进程，则其他进程可以附加该内存段。但是，大多数UNIX实现都禁止进程附加被标记为待删除的共享内存段 (SUSv3没有说明如何处理这种情况)。部分Linux应用程序依赖于该行为，这也是Linux的行为不同于大多数UNIX实现的行为的原因。

　　● IPC_STAT会将共享内存段关联的结构体shmid_ds的副本存储在参数buf指向的缓冲区中。

　　● IPC_SET会使用参数buf指向的缓冲区中的值更新共享内存段关联的结构体shmid_ds。

　　共享内存段可以将被锁定在RAM中，以确保它永远不会被换出，进而提升性能 (因为程序访问该页时不会出现缺页)。shmctl()的操作SHM_LOCK和SHM_UNLOCK分别用于将共享内存段锁定在内存中和解锁共享内存段 (允许它被换出)。SUSv3没有说明这2个操作，而且并非所有UNIX实现都提供了它们。

　　在Linux 2.6.10之前，只有特权进程 (能力CAP_IPC_LOCK) 才能将共享内存段锁定在内存中。从Linux 2.6.10开始，非特权进程也能锁定和解锁共享内存段，只要其有效用户ID匹配共享内存段的所有者或创建者的用户ID，并且该进程的资源限制RLIMIT_MEMLOCK足够大。

　　锁定共享内存段不会保证shmctl()调用完成时该内存段的所有页都驻留在内存中。相反，未驻留页只会在附加该内存段的进程通过后续引用将其缺页换入时才单独锁定。一旦被缺页换入，页会一直驻留在内存中 (即使所有进程都分离了该内存段)，直到被解锁 (即操作SHM_LOCK设置的是共享内存段的属性，而非调用进程的属性)。这里的缺页换入表示当进程引用未驻留在内存的页而导致缺页时，若交换区域中有对应的页，则会被重新加载到内存；否则 (第1次引用该页)，内核会分配新物理页，并更新该进程的页表和该内存段的记录数据结构。

　　mlock()也能用于锁定内存，但语义略有不同。

48.5 虚拟内存中共享内存的位置

　　对于x86-32架构，当将共享内存段附加到由内核选择的地址时，共享内存段会被附加到向上增长的堆和向下增长的栈之间的未被分配的空间。为了预留堆和栈的增长空间，共享内存段的附加从地址0x40000000开始。内存映射和共享库也位于该区域 (该默认位置会因内核版本和进程资源限制RLIMIT_STACK而有所差异)。

　　地址0x40000000由内核常量TASK_UNMAPPED_BASE定义。可以通过将该常量定义为其他值并重新构建内核来修改该地址 (可以在调用shmat()或mmap()时显式地指定地址来让共享内存段和内存映射的地址小于该常量)。

　　Linux特有的文件/proc/PID/maps可用于获取程序的共享内存段和共享库的位置。从内核2.6.14开始，可以通过文件/proc/PID/smaps来获取关于进程的所有映射的内存占用的更多信息。

　　● 第7~9行对应标准C库 (libc-version.so)。第12~14行对应动态链接器 (ld-version.so)。

　　● 第15行包含标签[stack]，对应进程的栈。

　　● 最后1行包含标签[vdso]。这是linux-gate虚拟动态共享对象的条目之一，该条目仅出现在内核2.6.12或更高版本。

　　● 第1列是一对由连字符分隔的数字，表示共享内存段映射的十六进制虚拟地址范围 (第2个数字是该内存段结尾的下一个字节的地址)。

　　● 第2列是保护和标志。前3个字母分别是读取、写入和执行保护 (连字符表示禁用保护)。最后1个字母是映射标志，可以为p (私有) 或s (共享)。

　　● 第3列是共享内存段在对应的映射文件内的十六进制偏移量 (以字节为单位)。对于System V共享内存段，该偏移量总是为0。

　　● 第4列是对应的映射文件所在设备的设备号 (主ID和次ID)。

　　● 第5列是映射文件的i-node号或System V共享内存段标识符。

　　● 第6列是共享内存段关联的文件名或其他标识符。对于System V共享内存段，该列由字符串SYSV和该内存段的键组成 (键IPC_PRIVATE的值是00000000)。字符串(deleted)是共享内存段实现导致的遗留问题。这种共享内存段被创建为不可见的tmpfs文件系统中的映射文件，然后被取消链接 (共享匿名内存映射也是以这种方式创建)。

48.6 将指针存储在共享内存中

48.7 共享内存限制

　　大多数UNIX实现都对System V共享内存施加了各种限制。

　　Linux施加了以下共享内存限制 (受限制影响的系统调用以及达到限制时产生的错误信息在括号中注明)：

　　● 限制SHMMNI是可以创建的共享内存段标识符的系统级限制 (shmget()和ENOSPC)。

　　● 限制SHMMIN是共享内存段的最小字节数。该限制固定为1，并且无法被修改。但是，共享内存段的实际最小字节数是系统页大小 (shmget()和EINVAL)。

　　● 限制SHMMAX是共享内存段的最大字节数，其实际上限取决于可用RAM和交换空间 (shmget()和EINVAL)。

　　● 限制SHMALL是共享内存段的页数总和的系统级限制。大多数UNIX实现没有该限制。该限制的实际上限取决于可用RAM和交换空间 (shmget()和ENOSPC)。

　　有些UNIX实现还施加了限制SHMSEG (Linux没有该限制)，该限制是可以附加的共享内存段数的进程级限制。

　　当系统启动时，共享内存限制会被设置为默认值。这些默认值可能因内核版本而异 (有些内核发行版使用不同于原版的默认值)。有些共享内存限制可以通过修改Linux特有的/proc文件系统来查看和修改。下表列出了这些限制对应的/proc文件 (基于x86-32架构)：

限制上限值/proc/sys/kernel中的对应文件
SHMMNI32768 (IPCMNI)shmmni
SHMMAX取决于可用内存shmmax
SHMALL取决于可用内存shmall

　　Linux特有的shmctl()操作IPC_INFO会检索结构体shminfo，该结构体包含各种共享内存限制的值。

　　Linux特有的shmctl()操作SHM_INFO会检索结构体shm_info，该结构体包含共享内存对象使用的系统资源的相关信息。
