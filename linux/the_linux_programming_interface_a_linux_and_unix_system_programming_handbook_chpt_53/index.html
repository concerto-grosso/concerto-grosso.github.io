《Linux/UNIX系统编程手册》第53章 POSIX信号量

发表于 2026-06-03 更新于 2026-06-04 分类于 Linux 本文字数： 5.3k

　　SUSv3详细说明了2种POSIX信号量：命名信号量和无名信号量。

　　● 命名信号量有1个名称。通过使用相同的名称调用sem_open()，不相关进程也能访问同一信号量。

　　● 无名信号量没有名称，它位于约定的内存位置。无名信号量可以由多个进程或线程共享。当多个进程共享无名信号量时，该信号量必须位于共享内存区域 (System V/POSIX共享内存或内存映射)。当多个线程共享无名信号量时，该信号量必须位于线程共享的内存区域 (例如，堆变量或全局变量)。

　　与System V信号量相同，POSIX信号量是值不能小于0的整数。若进程试图将信号量减少至小于0，则调用可能会阻塞进程或出错，具体取决于使用的函数。

　　某些系统没有提供POSIX信号量的完整实现。典型的限制之一是仅支持线程共享的无名信号量。Linux 2.4就是这种情况，而Linux 2.6以及提供NPTL的glibc库提供了完整的POSIX信号量实现。

　　对于带有NPTL的Linux 2.6，信号量操作 (增减) 被实现为基于系统调用futex()。

　　SUSv3没有说明如何实现命名信号量。对于有些UNIX实现，命名信号量被创建为标志文件系统中特定位置的文件。对于Linux，命名信号量被创建为名称格式为sem.name的小型POSIX共享内存对象，位于挂载在目录/dev/shm下的tmpfs文件系统中 (tmpfs文件系统具有内核持久性，所以命名信号量也就有内核持久性)。Linux从2.6开始支持命名信号量。

　　POSIX信号量和System V信号量都能用于同步进程行为。如前所述，POSIX IPC接口比System V IPC接口更简单，并且与传统UNIX文件模型更一致。相比System V信号量，POSIX命名信号量同样具有这样优点。

　　相比System V信号量，POSIX信号量还具有以下优点：

　　● POSIX信号量接口比System V信号量接口简单得多 (这种简洁性是在不牺牲功能的前提的下实现的)。

　　● POSIX命名信号量消除了System V信号量的初始化问题。

　　● 将POSIX无名信号量与动态分配的内存对象关联起来更简单：可以直接将无名信号量嵌入到内存对象。

　　● 对于信号量存在高度争用的情况 (即进程对信号量的操作经常因其他进程将该信号量设置为阻止该操作的值而被阻塞)，POSIX信号量和System V信号量的性能相似。但是，对于信号量争用较低的情况 (即进程对信号量的操作通常不会被阻塞)，POSIX信号量的性能明显优于System V信号量 (因为POSIX信号量仅在出现争用时才需要系统调用，而System V信号量总是需要系统调用)。

　　但是，相比System V信号量，POSIX信号量存在以下缺点：

　　● POSIX信号量的可移植性更差。

　　● POSIX信号量没有提供与System V信号量的撤销功能类似的功能。

　　POSIX信号量和Pthreads互斥锁都能用于同步 (同一) 进程的不同线程的操作，并且性能相似。但是，后者通常更可取，因为其所有权属性 (只有上锁的线程才能解锁) 能够强制实现良好的代码结构。相比之下，线程可以增加由其他线程减少的信号量。这种灵活性可能会导致结构不佳的同步设计 (所以信号量也称为并发编程的 "goto" 语句)。

　　对于多线程应用程序，有1种情况不能使用互斥锁：sem_post()是异步信号安全的，所以可以在信号处理函数中使用该函数来与其他线程同步，而操作互斥锁的Pthreads函数都是非异步信号安全的 (但是，由于更可取的方法通常是通过sigwaitinfo()等函数接收异步信号，而非使用信号处理函数，所以这一优点可能很少被需要)。

53.1 命名信号量

　　为了使用命名信号量，需要执行以下步骤：

　　● 函数sem_open()会创建并初始化信号量或打开现有信号量，然后返回句柄供后续调用使用。

　　● 函数sem_post()和sem_wait()分别会增加和减少信号量。

　　● 函数sem_getvalue()会检索信号量的当前值。

　　● 函数sem_close()会删除与调用进程与其之前打开的信号量的关联。

　　● 函数sem_unlink()会取消链接信号量并将其标记为待删除。

53.1.1 sem_open()

　　函数sem_open()会创建并打开新信号量或打开现有信号量，信号量由参数name标识。若成功，则返回sem_t指针，后续调用可以通过该指针来操作信号量。若出错，则返回SEM_FAILED (大多数实现都将SEM_FAILED 定义为(sem_t *)0或(sem_t *)-1，Linux将其定义为前者)。

　　参数oflag是决定打开现有信号量或创建并打开新信号量的位掩码。若该参数为0，则打开信号量。若该参数包含标志O_CREAT，并且名为name的信号量不存在，则创建信号量。若该参数包含标志O_CREAT和O_EXCL，并且名为name的信号量存在，则sem_open()会出错。

　　SUSv3没有说明oflag的任何访问模式标志 (O_RDONLY、O_WRONLY和O_RDWR)。很多UNIX实现和Linux默认以读写模式 (O_RDWR) 打开信号量，因为大多数使用信号量的应用程序都会调用sem_post()和sem_wait()。

　　若sem_open()用于打开信号量，则无需指定其他参数。若该函数用于创建信号量，则还需要指定参数mode和value (若名为name的信号量已存在，则这2个参数会被忽略)。

　　● mode是用于指定新信号量的权限的位掩码。可以指定给该参数的标志与文件权限标志 (见15.4) 相同，并且该参数也受限于进程umask。

　　● value是用于指定新信号量的值的无符号整数。信号量的创建和初始化是以原子方式执行。这避免了初始化System V信号量所需的复杂性。

　　SUSv3指出，试图操作sem_open()返回值指向的sem_t变量的副本会导致未定义的结果。换句话说，sem2的用法是不允许的。

　　fork()创建的子进程会继承父进程的所有打开的命名信号量引用。当fork()后，父子进程可以通过这些信号量同步操作。

53.1.2 sem_close()和sem_unlink()

　　当进程打开命名信号量时，系统会记录进程和信号量的关联。函数sem_close()会关闭 (终止) 这种关联，释放系统为该进程与信号量关联的所有资源，并减少信号量的进程引用计数。

　　当进程终止或调用exec()时，命名信号量会自动被关闭。

　　函数sem_unlink()取消链接名为name的信号量并将其标记为待删除 (一旦所有进程不再使用该信号量，它就会被删除)。

53.2 信号量操作

　　POSIX信号量在以下方面与System V信号量不同：

　　● 函数sem_post()和sem_wait()只会操作1个信号量。相比之下，System V系统调用semop()可以操作信号量集中的多个信号量。

　　● 函数sem_post()和sem_wait()只会将信号量加1和减1。相比之下，semop()可以将信号量加减任意值。

　　● POSIX信号量没有提供类似于System V信号量的等待信号量为0的操作。

　　System V信号量看似比POSIX信号量更全面，但情况并非如此。System V信号量所能实现的所有功能也能通过POSIX信号量实现。

53.2.1 sem_wait()

　　函数sem_wait()会将参数sem指向的信号量减1。

　　若信号量当前大于0，则sem_wait()会立即返回。若信号量当前为0，则该函数会阻塞进程，直到信号量大于0；届时，该函数会将信号量减1并返回。

　　若阻塞的sem_wait()调用被信号处理函数中断，则该调用会出错 (EINTR)，无论通过sigaction()注册信号处理函数时是否指定了标志SA_RESTART (对于有些UNIX实现，该标志会重启sem_wait())。

　　这个示例展示了函数sem_wait()的命令行界面。

　　函数sem_trywait()是sem_wait()的非阻塞版本。若减少操作不能立即完成，则该函数会出错 (EAGAIN)。

　　函数sem_timedwait()与sem_wait()完全相同，但可以通过参数abs_timeout指定该调用的阻塞时间上限。若减少操作未在阻塞时间耗尽之前完成，则该函数会出错 (ETIMEOUT)。

　　abs_timeout通过自纪元以来的以秒和纳秒为单位的绝对值来指定超时时间。为了设置相对超时时间，可以通过clock_gettime()获取CLOCK_REALTIME时钟的当前值并加上所需的值来获取合适的结构体timespec。

　　sem_waitdwait()源于POSIX.1d (1999)，但并非所有UNIX实现都提供了该函数。

53.2.2 sem_post()

　　函数sem_post()会将参数sem指向的信号量加1。

　　若在调用sem_post()之前信号量的值为0，并且某个进程 (或线程) 被阻塞在减少该信号量，则该进程 (或线程) 会被唤醒，其sem_wait()调用会继续减少信号量。若多个进程 (或线程) 被阻塞在减少该信号量，并且这些进程 (或线程) 都运行在默认的轮询分时调度策略下，则哪个进程 (或线程) 会被唤醒并减少信号量是不确定的。

　　SUSv3指出，若多个进程或线程运行在实时调度策略下，则等待时间最长且优先级最高的进程或线程将被唤醒。

　　与System V信号量相同，增加POSIX信号量相当于释放关联的共享资源。

53.2.3 sem_getvalue()

　　函数sem_getvalue()会将参数sem指向的信号量的值存储在参数sval指向的int中。

　　若多个进程 (或线程) 被阻塞在减少信号量，则sval中的值将取决于实现。SUSv3允许该值为0或当前阻塞在sem_wait()的进程数的相反数。Linux和有些UNIX实现采用前者的行为，而部分其他UNIX实现采用后者的行为。

　　当存在阻塞在减少信号量的进程时，sval返回的负值可能有用 (尤其是对于调试)。但是，SUSv3不要求这种行为，因为某些系统实现POSIX信号量的技术无法记录当前阻塞在减少信号量的进程数。

　　当sem_getvalue()返回时，sval中的值可能已经过时。根据该信息执行后续操作的程序可能受到TOCTTOU竞争条件的影响。

53.3 无名信号量

　　无名信号量是存储在应用程序分配的内存的sem_t变量 (也称为基于内存的信号量)。通过将信号量放置在进程或线程共享的内存区域中，使其可供这些进程或线程使用。

　　sem_wait()、sem_post()和sem_getvalue()等用于操作命名信号量的函数也能用于无名信号量。此外，使用无名信号量还需要函数sem_init()和sem_destroy() (这2个函数不应该与命名信号量一起使用)。

　　使用无名信号量可以避免为信号量命名。这在以下情况很有用：

　　● 线程之间共享的信号量不需要名称。将无名信号设置为共享变量 (全局变量或堆变量) 会自动使其对所有线程可访问。

　　● 相关进程之间共享的信号量不需要名称。若父进程在共享内存区域 (例如，共享匿名映射) 中分配无名信号量，则子进程会继承该区域，所以该信号量会成为fork()调用的一部分。

　　● 当需要构建动态数据结构 (例如，二进制树)，并且其中的每个项会关联1个信号量时，最简单的实现方法是为每个项分配1个无名信号量。为每个项创建有名信号量需要设计用于命名信号量和管理这些名称的方案。

53.3.1 sem_init()

　　函数sem_init()会将参数sem指向的信号量初始化为值value。参数pshared指定信号量是在线程间共享还是在进程间共享。

　　● 若pshared为0，则表示在调用进程的所有线程之间共享信号量。在这种情况下，sem通常是全局变量或堆变量的地址。线程共享的信号量具有进程持久性 (当进程终止时，它回自动被销毁)。

　　● 若pshared不为0，则表示在进程之间共享信号量。在这种情况下，sem必须是共享内存区域中的某个位置的地址。信号量的持久性就是其所处内存区域的持久性。fork()创建的子进程会继承父进程的内存映射，所以子进程会继承进程共享的信号量，父子进程可以通过这种信号量来同步行为。

　　参数pshared是必要的，原因如下：

　　● 有些实现不支持进程共享的信号量。对于这些系统，将该参数指定为0会导致sem_init()出错。直到Linux 2.6和NPTL实现出现之前，Linux都不支持进程共享的无名信号量 (sem_init()的LinuxThreads实现会在pshared不为0时导致错误ENOSYS)。

　　● 对于支持进程共享的信号量和线程共享的信号量的实现，系统需要根据共享类型来执行对应的特殊行为 (某些系统可能会根据共享类型执行优化)。

　　sem_init()的NPTL实现会忽略pshared，因为这2种共享类型都不需要特殊操作。尽管如此，为了保证应用程序可移植性和前瞻性，应该为pshared指定为合适的值。

　　sem_init()的SUSv3规范要求它在出错时返回-1，但没有说明成功时的返回值。尽管如此，对于大多数现代UNIX实现的手册，该函数在成功时返回0 (Solaris手册中的该返回值与SUSv3规范类似，但根据OpenSolaris的源代码，该函数在成功时会返回0)。SUSv4对此进行了修正，明确规定该函数在成功时应该返回0。

　　无名信号量没有权限设置 (sem_init()没有与sem_open()的参数mode类似的参数)。无名信号量的访问控制由底层的共享内存区域决定。

　　SUSv3指出，初始化已初始化的无名信号量或对指定给sem_init()的参数sem的sem_t变量的副本执行操作会导致未定义的行为。

53.3.2 sem_destroy()

　　函数sem_destroy()会销毁sem指向的无名信号量。当且仅当没有进程或线程等待无名信号量时，才能安全地销毁它。当无名信号量被sem_destroy()销毁后，可以通过sem_init()重新初始化。

　　无名信号量的销毁应该先于底层内存的释放。若无名信号量是自动分配的变量，则应该在对应的函数返回之前销毁该信号量。若无名信号量存储在POSIX共享内存区域，则应该在所有进程都不再使用该信号量之后和取消链接该共享内存对象之前释放该信号量。

　　对于某些实现，不调用sem_destroy()不会导致问题。但是，对于其他实现，未成功调用该函数会导致资源泄露。可移植应用程序应该调用该函数来避免此类问题。

53.4 信号量限制

　　SUSv3定义来2个POSIX信号量限制：SEM_NSEMS_MAX和SEM_VALUE_MAX。

　　● 限制SEM_NSEMS_MAX定义了进程可以拥有的POSIX信号量的最大数量。SUSv3要求该限制至少为256。对于Linux，POSIX信号量数仅受限于可用内存。

　　● 限制SEM_VALUE_MAX定义了POSIX信号量的最大值。SUSv3要求该限制至少为32767，Linux允许该限制最大为INT_MAX。
